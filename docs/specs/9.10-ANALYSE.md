## 0. Reformulation et périmètre

### Objectif
Instrumenter l'entonnoir de conversion (visiteur → analyse lancée → analyse
aboutie → export → affinage → feedback) avec des events **anonymes, agrégés,
first-party**, dans l'esprit de minimisation de la table `Feedback`. Cible :
fermer l'item #1 « CRITIQUE » du FORECAST §10 (trafic réel inconnu) et alimenter
le tableau de bord d'adoption FORECAST §7.2.

### In (à analyser, confirmé par le code)
- **A1.** Table SQLite `events` (esprit `Feedback` : aucune IP, aucune PII,
  bands / enums / booléens seulement).
- **A2.** Endpoint `POST /events` anonyme, public, status 201, calqué sur
  `POST /feedback` (`backend/app/main.py:318-345`). Rate-limité via la brique 9.9.
- **A3.** Émission front fire-and-forget calquée sur
  `frontend/lib/api.ts::sendFeedback` (api.ts:97-111), branchée aux handlers
  réels de `frontend/app/page.tsx` + un beacon client pour `frontend/app/methode/page.tsx`.

### Out (hors périmètre, acté)
- Tout SaaS analytics externe (GA / Plausible / Umami). On reste **first-party**
  (confirmé reco Q1 de 9.9).
- **Dashboards / rapports d'exploitation** des events (lecture, corrélations) =
  livrable DISTINCT ultérieur, comme les « rapports d'exploitation » de 9.7.
- Auth, comptes, cookies de session identifiants (bloqués, cf. CONTEXT §9.6).
- **Rate-limit `/analyze`** : déjà livré en 9.9
  (`app/rate_limit.py`, appliqué main.py:264). 9.10 ne fait que **réutiliser** la
  fabrique `rate_limiter` sur `/events`.

### Ce qui a déjà bougé depuis le pré-cadrage 9.9
Confirmé dans le code (le pré-cadrage du prompt parlait du futur ; voici l'état réel) :
- `app/rate_limit.py` **existe** et expose `rate_limiter(limit, window_seconds)`
  (dépendance FastAPI, 429 + `Retry-After`, IP via `Fly-Client-IP` jamais
  persistée). Déjà appliqué à `/analyze` (limit=10/60s, main.py:264) **et** à
  `/feedback` (limit=60/60s, main.py:321). Donc rate-limiter `/events` = une ligne
  `Depends(rate_limiter(...))`, brique prête.
- L'isolation de test de l'état rate-limit est en place (`conftest.py:54-68`,
  fixture autouse).

## 1. Confrontation au code réel (fichier:ligne)

### 1.1 Table `events` — mécanisme de création (`backend/db/session.py`)

`init_db()` (session.py:58-61) fait `Base.metadata.create_all` puis
`_migrate_comparables` (session.py:51-55) qui ne gère QUE l'ajout de colonnes à
`comparables` (boucle `_ADD_COLUMNS`, session.py:35-48). Conséquences vérifiées :

- Une nouvelle classe SQLAlchemy `Event` dans `db/models.py` (à côté de
  `Feedback`, models.py:54-70) est créée automatiquement par `create_all` au
  prochain `init_db` (`@app.on_event("startup")`, main.py:51-53). **Zéro DDL
  manuelle** — exactement comme `feedback` (créée sans entrée dans `_ADD_COLUMNS`).
- `create_all` est idempotent : aucun risque sur la base prod `/data/comparables.db`.

**Statut sur `_migrate_events` (YAGNI)** : `_migrate_comparables` n'existe que
parce que `comparables` a gagné des colonnes APRÈS sa création initiale en prod
(dpe, postal_code, confort…). Pour `events`, la table sera créée **complète dès le
départ** ; `create_all` suffit. Un `_migrate_events` n'apporterait rien
aujourd'hui. **Reco : ne PAS créer `_migrate_events` maintenant** (YAGNI). Le
créer seulement le jour où on ajoute une colonne à `events` sur une base prod
existante. C'est la même conclusion que 9.9-ANALYSE §1.2.

> Conséquence de design qui découle de ce choix : pour éviter d'avoir à migrer
> plus tard, le schéma doit être pensé **stable et générique** dès maintenant
> (voir §2.2 : un champ `props` JSON générique évite d'ajouter une colonne par
> nouvel event — mais ouvre une surface PII, d'où le garde-fou allowlist).

### 1.2 `POST /events` — où ça s'insère (`backend/app/main.py`)

Le gabarit `POST /feedback` (main.py:318-345) est le modèle exact :
- public, `status_code=201`,
- ouvre/ferme `SessionLocal()` en `try/finally` (main.py:323-336),
- **ne journalise jamais** le champ libre (`feedback_logger.info` ne loggue que
  `rating`, `bool(comment)`, `analysis_id` — jamais le texte, main.py:338-344),
- rate-limité par `Depends(rate_limiter(limit=60, window_seconds=60))` (main.py:321).

`/events` se calque :
- Modèle Pydantic `EventIn` à côté de `FeedbackIn` (main.py:78-84).
- Import `Event` ajouté à `from db.models import Comparable, Feedback` (main.py:14).
- Logger dédié `events_logger = logging.getLogger("events")` (cf. `feedback_logger`
  main.py:27) — à ajouter à la liste des loggers nommés de `backend/CLAUDE.md` §12
  (qui liste déjà `rate_limit` depuis 9.9).
- `Depends(rate_limiter(...))` pour `/events` (voir Q6).

**Héritage CORS — vérifié, rien à faire** : le middleware global (main.py:41-48)
pose `allow_origin_regex=r"https://.*\.vercel\.app"` + localhost,
`allow_methods=["*"]`, `allow_headers=["*"]`. `/events` en hérite automatiquement.
L'appel cross-origin Vercel → Fly fonctionne sans config supplémentaire.

`sendFeedback` (api.ts:97-111) est déjà le modèle fire-and-forget : `try/catch`

remontée). `page.tsx` est `"use client"` (page.tsx:1) : les hooks sont triviaux à

**Garde-fous fire-and-forget vérifiés** :
- Ne JAMAIS `await` le `sendEvent` dans le chemin critique de `handleAnalyze` /
  `handleRefine` : l'event ne doit ni retarder ni faire échouer l'analyse (le
  `try/catch` de `analyzeListing` ne doit voir QUE l'erreur d'analyse).
- `keepalive: true` sur le `fetch` (ou `navigator.sendBeacon`) utile seulement pour
  un futur event « au unload » ; pour les events in-session (la liste 9.10), un
  fetch non-awaité suffit, `keepalive` = ceinture bon marché.

### 1.4 `feedback_submitted` : ne pas le router via `/events`

État réel : le feedback est **déjà** persisté en table `feedback` via
`POST /feedback` (main.py:318, `sendFeedback` api.ts:97, `handleFeedback`
page.tsx:543). La donnée `rating` + `global_score` + `verdict` + `prompt_variant`
y est déjà. Router `feedback_submitted` aussi vers `/events` créerait **deux
sources de vérité** pour le même fait → écarts de comptage.
- **Reco : (a) garder `/feedback` seul** ; lire la table `feedback` quand on aura
  besoin de ce signal. C'est la même reco que 9.9-ANALYSE §1.4 / Q7. Zéro
  duplication. → confirmé en Q7 ci-dessous.

### 1.5 `methode_view` : beacon client sur un server component

Vérifié : `frontend/app/methode/page.tsx` est un **server component** — il exporte
`metadata` (methode/page.tsx:6-10) et n'a **pas** de `"use client"`. Émettre un
event au montage y impose un **petit composant client dédié** monté dans la page
(`<PageViewBeacon name="methode_view" />`), PAS de passer la page en client (ce qui
casserait l'export `metadata` et le SSR/SEO). Le beacon est un composant
`"use client"` minimal avec un `useEffect([])` qui appelle `sendEvent("methode_view")`.
**Reco : beacon client minimal**, zéro impact sur le rendu serveur. Même approche
réutilisable pour `page_view` si on veut factoriser, mais `page.tsx` étant déjà
client, un `useEffect` inline y suffit.

### 1.6 Wording RGPD existant à confronter

Deux endroits affirment au visiteur que rien n'est conservé :
- `frontend/components/design/Footer.tsx:26` : « Analyses non conservées ».
- `frontend/app/methode/page.tsx:154` : « Vos analyses ne sont pas conservées. »

Ces phrases visent le **contenu d'annonce** (vrai : `/analyze` ne persiste rien,
les photos sont « analysées en transit » page.tsx:445). Collecter des **events
anonymes agrégés** ne stocke pas l'annonce → ne contredit pas stricto sensu
« analyses non conservées ». **Mais** c'est limite, et un utilisateur attentif
pourrait l'opposer. Deux cas :
- **Sans cookie** (events anonymes purs) : le wording actuel reste défendable ;
  une micro-mention « mesure d'audience anonyme » est prudente mais pas obligatoire.
- **Avec cookie de session** (Q3 option b) : il faut **mettre à jour le wording**
  (micro-mention, comme la micro-mention RGPD ajoutée en 9.7). → lié à Q2/Q3.

## 2. RGPD — ce qui peut devenir une PII (le cœur de 9.10)

Posture du projet = **minimisation assumée** (modèle `Feedback` : « ni IP, ni
identifiant, ni extrait d'annonce », models.py:56-57). 9.10 doit la tenir.
Points de vigilance du plus au moins risqué :

### 2.1 `referrer` complet → ne stocker que le domaine
Un referrer brut peut porter des paramètres identifiants (UTM, tokens, parfois un
email en query string sur un lien mal fait). **Garde-fou : ne stocker que le
domaine** (`new URL(document.referrer).hostname`), jamais le path ni la query.
FORECAST §7.1 le précise déjà (« domaine seul »). À verrouiller par test (type 9.7).
Idem `path` de `page_view` : c'est le path **interne du site** (`/`, `/methode`),
pas l'URL d'annonce — sûr, mais à borner à une allowlist de chemins connus pour
ne jamais laisser fuir un path inattendu.

### 2.2 Champ `props` libre → risque de PII injectée (point dur n°2 du prompt)
Le choix d'un champ `props` JSON générique (pour éviter `_migrate_events` à chaque
nouvel event, §1.1) **ouvre une surface PII** : un dev pressé pourrait y mettre
l'adresse, l'URL d'annonce ou un extrait de texte. Garde-fous possibles, du plus
strict au plus souple :
- **(i) Validation Pydantic stricte par type d'event** : un modèle discriminé
  (`EventIn` avec `name` ∈ allowlist + props typées par event). Le plus sûr : le
  serveur **refuse** (422) tout event hors allowlist et toute prop non déclarée.
  Coût : un peu plus de code (un schéma par event), mais c'est la garantie forte.
- **(ii) Allowlist de noms d'events + props en colonnes typées dédiées** (pas de
  JSON libre) : `mode`, `format`, `reason`, `from_scope`, `to_scope`, `score_band`,
  `confidence`, `pillar_price_status`, `referrer_domain`, `path`, `address_entered`
  (bool). Aucune colonne « texte libre ». Le plus minimal côté PII, mais ajoute des
  colonnes (donc rapproche du besoin `_migrate_events` futur si on en ajoute).
- **(iii) `props` JSON libre + filtre serveur (allowlist de clés)** : souple mais
  c'est exactement la porte par laquelle une PII passe. **Écarté** sauf si couplé à
  une allowlist de clés stricte côté serveur (qui revient à (i)/(ii)).

→ Arbitrage remonté en **Q2**. Reco analyste : **(i) ou (ii)** — schéma fermé,
jamais de texte libre dans `events`. C'est l'invariant qui protège l'esprit
`Feedback`.

### 2.3 Empreinte de session anonyme (`repeat_session`, volume/session) — point le plus délicat
FORECAST §7.1 liste `repeat_session` et `analysis_volume_per_session` comme proxys
de rétention et de **signal B2B** (1 user, 20 analyses = pro). Mesurer ça impose de
**corréler plusieurs requêtes d'un même navigateur**. Trois familles :

- **(a) Aucun identifiant** : events purs sans corrélation inter-requêtes. On peut
  garder un compteur **en mémoire d'onglet** (variable JS de session de navigation,
  perdue au refresh) pour un `analysis_volume_per_session` approximatif intra-page,
  mais **pas** de `repeat_session` fiable (pas de mémoire entre visites). **Zéro
  cookie / zéro consentement / zéro bandeau.**
- **(b) Cookie / `localStorage` 1ʳᵉ partie avec UUID aléatoire opaque** (pas dérivé
  de l'IP/UA) : techniquement un identifiant, mais **non-PII** (désigne un
  navigateur, pas une personne) et **1ʳᵉ partie non publicitaire**. Permet
  `repeat_session` et le proxy B2B fiables. La CNIL **exempte de consentement** les
  cookies de **mesure d'audience** strictement nécessaires SI configurés en
  conformité (pas de croisement inter-sites, finalité limitée, durée de vie bornée,
  pas de partage tiers). Mais ce n'est **pas** « zéro cookie » : impose une
  micro-mention et la MAJ du wording « non conservées » (§1.6).
- **(c) Fingerprint douce** (hash UA + langue + …) : **à écarter** — la CNIL la
  traite comme un traceur soumis à consentement, et c'est plus intrusif qu'un
  cookie 1ʳᵉ partie. Écartée.

**Pour/contre RGPD synthétique :**

| Option | Bandeau requis ? | Signal funnel couvert | Surface RGPD |
|---|---|---|---|
| (a) sans cookie | **Non** | funnel complet SAUF `repeat_session` fiable et volume/session cross-visite | minimale (reste sous le seuil traceur) |
| (b) cookie 1ʳᵉ partie UUID | **Non si exemption mesure d'audience respectée** (sinon oui) | funnel complet + rétention + proxy B2B | identifiant non-PII + micro-mention + MAJ wording |
| (c) fingerprint | Oui | idem (b) | la plus intrusive, écartée |

→ Arbitrage structurant remonté en **Q3**. C'est **le** point qui conditionne Q1
(bandeau).

### 2.4 `global_score` en band, pas en valeur exacte
FORECAST §7.1 dit « band » pour `analysis_succeeded`. Une valeur exacte n'est pas
une PII, mais rester en bands (ex. `<40 / 40-59 / 60-79 / 80+`) est cohérent avec
la minimisation et suffit au funnel. **Reco : bands.**

### 2.5 IP — déjà cadrée par 9.9, ne jamais l'écrire dans `events`
Le rate-limit (`app/rate_limit.py`) utilise l'IP comme **clé de dict en mémoire
transitoire, jamais journalisée, jamais persistée** (rate_limit.py:5-6, 37-50).
**Ligne rouge pour 9.10** : ne JAMAIS écrire d'IP dans la table `events`, ni dans
un log d'event, ni dans la DB. À verrouiller par revue + test (l'`events_logger`
ne loggue jamais d'IP, comme `rate_limit` logger et `feedback_logger`).

### 2.6 Consentement / bandeau — synthèse
Si la collecte est **anonyme, agrégée, first-party, sans cookie identifiant et IP
non persistée** (voie (a) de Q3), la doctrine CNIL **n'impose pas** de consentement
ni de bandeau (mesure d'audience exemptée / absence de traceur). Si on pose un
cookie de session (Q3 option b), il faut vérifier qu'il rentre dans l'exemption
« mesure d'audience » CNIL pour rester sans bandeau ; sinon bandeau requis.
**Reco analyste : voie sans cookie (a) pour la v1**, pour ne pas ouvrir ce débat
sur un MVP < 1 €/mois. → Q1/Q3.

## 3. Pièges techniques (points durs)

### 3.1 `llm_fallback` invisible côté front → event SERVEUR, pas client
**Vérifié dans le code** : le fallback LLM est émis en **un seul point**,
`backend/app/llm_semantic.py:235-242` — sur exception, `analyze_semantic` loggue
`"LLM call failed"` (ERROR, logger `llm_semantic`) puis **retourne
`dict(_FALLBACK)`** (un dict d'analyse « normale » : transparency_score=50,
verdict « Moyenne »…, llm_semantic.py:114-118). `run_full_analysis` renvoie alors
un **200 normal** : le front reçoit une analyse réussie et **ne peut pas** savoir
qu'il s'agit d'un fallback. Donc :
- Le front ne peut émettre `analysis_failed(reason=llm_fallback)` de façon fiable.
- Deux options : **(a)** mesurer `llm_fallback` **côté serveur** (le seul qui le
  connaît) — un INSERT d'event au point unique du fallback (llm_semantic.py:242),
  cohérent avec 9.3 qui a déjà verrouillé le marqueur `LLM call failed` ; **(b)**
  faire remonter un flag `degraded` dans `AnalyzeResponse` → **change le contrat
  `/analyze`** (anti-pattern §11.9 / CLAUDE), refusé.
- **Reco : (a)**. Implique que `analysis_failed(reason=llm_fallback)` est un event
  **serveur**, tandis que `url_unreachable` / `no_input` restent côté client (le
  front les voit via les codes 422/400). → Q4. C'est aussi la reco de 9.9-ANALYSE §3.

> Note d'implémentation : un INSERT d'event depuis `llm_semantic.py` couple ce
> module à la couche DB (aujourd'hui il ne connaît que le cache mémoire). Plus
> propre : émettre le fallback depuis `analysis.run_full_analysis` (qui appelle déjà
> la DB) en exposant un signal interne (ex. un drapeau dans le dict retourné par
> `analyze_semantic`, NON exposé dans la réponse HTTP). À cadrer en SPEC.

### 3.2 `page_view` réveille la VM auto-stop — borné, à acter
L'auto-stop Fly se réveille sur **toute** requête entrante, y compris
`POST /events`. La leçon « ne pas réveiller la VM » (CONTEXT §9.3) vise le
**polling périodique** (cron/setInterval), pas un event émis **sur action
utilisateur**. `page_view` réveille la VM dès l'arrivée d'un visiteur (avant même
`/analyze`) : léger surcoût (réveils sur visiteurs qui ne lancent pas d'analyse),
mais **borné par le trafic réel** (FORECAST §5.2 : base M12 ~700 visiteurs/mois ;
réveiller une VM est gratuit, seul le temps CPU d'un INSERT compte, négligeable).
< 1 €/mois reste tenu. → Q5 : `page_view` serveur (reco) vs client-only.

### 3.3 Volume / rétention de la table `events` (point dur n°3)
`page_view` est l'event le plus volumineux (1+ par visiteur ; plusieurs events par
session active). Ordre de grandeur FORECAST : base M12 ~700 visiteurs/mois,
ambitieux ~3000/mois → quelques milliers de lignes/mois au pic. SQLite mono-writer
encaisse trivialement (le mono-writer ne devient un sujet que « > quelques req/s »,
FORECAST §5.3 — très loin). **Aucun problème de volume** à l'horizon du modèle, sur
le volume Fly 1 Go.
- **Rétention/purge dès le départ ou YAGNI ?** Reco analyste : **YAGNI sur l'auto-purge**
  (pas de cron de purge maintenant : le volume est négligeable et un cron de purge
  réveillerait la VM, à rebours de l'auto-stop). MAIS prévoir **dès la SPEC** la
  possibilité d'une purge **manuelle** (un endpoint admin `X-Admin-Token` type
  `/admin/comparables/maintenance`, ou un simple `DELETE WHERE created_at < …`
  lançable en SSH) pour pouvoir tronquer si le besoin apparaît. Documenter la
  politique (« events conservés tant que le volume le permet, purge manuelle
  disponible ») plutôt que d'automatiser prématurément. → Q5bis.

### 3.4 Double-comptage (point dur n°5 : qualité de la mesure)
- **React StrictMode (dev)** monte les composants **deux fois** → un `useEffect([])`
  de `page_view` émettrait 2 events en dev. En prod (build Vercel) StrictMode ne
  double pas les effets, donc l'impact est limité au dev. Garde-fou propre : un
  `useRef` « déjà émis » dans le beacon/effect pour idempotence intra-montage.
  À acter dans la SPEC (sinon les chiffres de dev sont faux, pas la prod).
- **`handleRefine` rejoue `/analyze`** (page.tsx:506-521) : il ne faut PAS que
  chaque refine compte comme un nouveau `analysis_started` (sinon `taux_lancement`
  faussé). Reco : `district_refine` est son **propre** event ; `analysis_started`
  n'est émis **que** dans `handleAnalyze` (page.tsx:485), jamais dans `handleRefine`.
  Idem `analysis_succeeded` : une ré-analyse de refine ne doit pas regonfler le
  compteur d'aboutissements (à trancher en SPEC : soit on ne ré-émet pas
  `analysis_succeeded` sur refine, soit on le distingue par une prop).
- **Events perdus (fire-and-forget = best-effort)** : un fetch non-awaité peut être
  abandonné (onglet fermé, réseau coupé). **Acceptable** pour de la **tendance**
  (le funnel mesure des ordres de grandeur et des ratios, pas une comptabilité
  exacte). À **acter explicitement** dans la SPEC : les compteurs events sont des
  proxys de tendance, pas des totaux exacts.

### 3.5 `prompt_variant` — hors périmètre
Listé FORECAST §7.1 mais **9.8 différé** et le front ne l'envoie pas encore
(CONTEXT §9.8). Ne pas chercher à le câbler ici : dépendance non satisfaite.

## 4. Dépendances et ordre

- **9.10 ne dépend de rien de bloquant.** Tous les prérequis sont en place :
  - pattern `/feedback` livré (9.7),
  - brique `rate_limiter` livrée (9.9, `app/rate_limit.py`) → réutilisable sur `/events`,
  - handlers front existent (page.tsx), `sendFeedback` = modèle (api.ts),
  - table créée par `create_all` (session.py), CORS déjà bon (main.py:41-48),
  - isolation de test en place (conftest.py).
- **Ordre interne recommandé** : (1) table `events` + `POST /events` + rate-limit +
  tests backend (verrous : pas de PII, allowlist de noms, IP jamais écrite) ;
  (2) `sendEvent` + branchements front + beacon `methode_view` ; (3) event serveur
  `llm_fallback` ; (4) wording RGPD si Q3 = cookie.
- **Prérequis externe** : **aucun** (pas de domaine, pas de vendor, pas de secret,
  pas de DNS — contrairement à 9.2/9.6). Seule dépendance « humaine » = trancher les
  arbitrages GATE 1 ci-dessous (surtout Q1/Q3 RGPD).
- **Aval** : 9.10 est **prérequis** de la quasi-totalité des décisions du FORECAST
  (§6.1 #1 — choix B2C/B2B via le signal volume/session, justification d'un budget
  paid via un CAC mesurable, priorisation SEO). Aucune de ces suites n'est dans 9.10.

## 7. QUESTIONS POUR L'HUMAIN (GATE 1)

> Seuls les arbitrages **structurants** (RGPD, stockage, cookie, vendor, périmètre)
> sont listés. Le reste (nom des bands, libellé exact des enums, garde StrictMode)
> est de la SPEC, pas de la GATE.

1. **RGPD — bandeau / consentement nécessaire ?**
   - Options : (a) **non** — events anonymes, agrégés, first-party, **sans cookie
     identifiant**, IP non persistée → exemption mesure d'audience CNIL, pas de
     bandeau ; (b) **oui** — uniquement si on pose un cookie non exempté (dépend de Q3).
   - **Reco : (a)**, à condition de retenir la voie « sans cookie » en Q3. Évite
     d'ouvrir le débat bandeau sur un MVP < 1 €/mois.

2. **Schéma de la table `events` et garde-fou anti-PII dans les props.**
   - Options : (i) **schéma discriminé strict** (`name` ∈ allowlist + props typées
     par event, refus 422 hors allowlist) ; (ii) **colonnes typées dédiées** (mode,
     format, reason, from/to_scope, score_band, confidence, pillar_price_status,
     referrer_domain, path, address_entered bool — **aucun** texte libre) ;
     (iii) `props` JSON libre + filtre serveur (**écarté** seul : porte à PII).
   - **Reco : (i) ou (ii) — schéma fermé, jamais de texte libre.** C'est
     l'invariant qui empêche une adresse / URL / extrait d'entrer dans `events`.
     (ii) est le plus lisible ; (i) le plus extensible. À trancher selon le goût
     entre lisibilité SQL et souplesse d'ajout d'events.

3. **Session anonyme (`repeat_session` / volume-par-session, proxy B2B) :**
   - Options : (a) **rien** — compteur en mémoire d'onglet seulement (perdu au
     refresh), zéro cookie / zéro consentement ; pas de `repeat_session` fiable ;
     (b) **cookie / localStorage 1ʳᵉ partie** avec UUID aléatoire opaque non-PII —
     proxy rétention + B2B fiables, mais introduit un identifiant (vérifier
     exemption CNIL, impose micro-mention + MAJ wording « non conservées ») ;
     (c) **fingerprint** — **écartée** (traceur soumis à consentement).
   - **Reco : (a) pour la v1** (zéro friction RGPD, débloque déjà tout le haut/milieu
     du funnel — la vraie inconnue #1 du FORECAST). Passer à (b) **seulement** si le
     **signal B2B** (volume/session récurrent) devient un objectif explicite et qu'on
     accepte la micro-mention + MAJ wording. C'est **le** choix qui conditionne Q1.

4. **`llm_fallback` mesuré côté serveur (pas client), sans toucher le contrat `/analyze` ?**
   - Le fallback est invisible côté front (`run_full_analysis` renvoie 200 normal,
     `_FALLBACK` servi comme une analyse réussie — vérifié llm_semantic.py:242).
   - Options : (a) **oui** — event serveur au point unique du fallback (le client ne
     gère que `url_unreachable` / `no_input`) ; (b) flag `degraded` dans
     `AnalyzeResponse` → **change le contrat** (anti-pattern §11.9), refusé.
   - **Reco : (a)** — émis idéalement depuis `run_full_analysis` (déjà couplé DB),
     pas depuis `llm_semantic` (cache-only aujourd'hui).

5. **`page_view` côté serveur (réveille la VM à chaque visiteur) ou strictement client ?**
   - Options : (a) `POST /events` à l'arrivée — donnée fiable mais réveille la VM même
     pour les visiteurs sans analyse (surcoût borné, négligeable, §3.2) ; (b) ne
     compter que les visiteurs qui interagissent — sous-compte le haut du funnel
     (numérateur de `taux_lancement` faussé).
   - **Reco : (a)** — le haut du funnel (visiteurs uniques) est précisément
     l'inconnue #1 du FORECAST (H5) ; le surcoût de réveil est négligeable.

   **5bis. Rétention des events : auto-purge dès le départ ou purge manuelle (YAGNI) ?**
   - Options : (a) **YAGNI** — pas de cron de purge (volume négligeable, un cron
     réveillerait la VM) ; prévoir seulement une **purge manuelle** (endpoint admin
     `X-Admin-Token` ou `DELETE` en SSH) si besoin ; (b) cron de purge périodique
     (rétention N jours) dès maintenant.
   - **Reco : (a) YAGNI + purge manuelle disponible**, politique documentée. Le
     volume FORECAST (quelques k lignes/mois) ne justifie pas d'automatiser.

6. **Rate-limiter `/events` (et confirmer `/feedback`) ?**
   - Un nouvel endpoint d'écriture public sans limite recrée la surface d'abus
     qu'on a fermée sur `/analyze` (9.9). La brique `rate_limiter` est prête.
   - Options : (a) **oui, large** — `Depends(rate_limiter(...))` sur `/events`, seuil
     généreux car plusieurs events par session légitime (ex. 120/min/IP) ; (b) non.
   - **Reco : (a) un rate-limit large** (à dimensionner pour ne PAS bloquer un
     visiteur normal qui émet ~5-10 events par session, tout en coupant le spam
     d'INSERT). `/feedback` est déjà à 60/min (main.py:321), confirmer.

7. **`feedback_submitted` : via `/events` ou garder `/feedback` seul ?**
   - Options : (a) **garder `/feedback`** et lire la table `feedback` pour ce signal
     (zéro duplication, donnée déjà collectée par 9.7) ; (b) émettre aussi un event
     léger (deux sources de vérité → écarts de comptage) ; (c) tout migrer dans
     `events` (casse 9.7, écarté).
   - **Reco : (a).** La donnée existe déjà en table `feedback`.

8. **Périmètre v1 : quels events des 11 du FORECAST §7.1 on livre maintenant ?**
   - `repeat_session` et `analysis_volume_per_session` cross-visite dépendent de Q3
     (cookie). Si Q3 = (a) sans cookie, **ces deux events ne sont pas fiables** en v1.
   - Options : (a) **v1 = funnel sans corrélation de session** (page_view,
     methode_view, analysis_started/succeeded/failed, report_export, district_refine,
     address_entered ; feedback_submitted reste sur `/feedback`) — couvre l'inconnue
     #1 (trafic + taux du funnel) ; le signal B2B/rétention attend une v2 si Q3 bascule
     en cookie ; (b) v1 complète avec cookie (impose Q3=b, Q1 peut basculer).
   - **Reco : (a)** — livrer le funnel anonyme d'abord (zéro friction RGPD, débloque
     le FORECAST), garder `repeat_session`/volume-session pour une v2 conditionnée à
     Q3. Acter explicitement que les compteurs sont des **proxys de tendance**
     (fire-and-forget best-effort, §3.4), pas une comptabilité exacte.
