## 1. Objectif et perimetre

### Objectif
Instrumenter l'entonnoir de conversion (visiteur -> analyse lancee -> analyse
aboutie -> export -> affinage -> feedback) avec des events anonymes, agreges,
first-party (esprit de minimisation de la table `Feedback`), pour remplacer les
hypotheses de trafic du FORECAST par des faits mesures, sans cookie, sans PII,
sans toucher le contrat `/analyze`.

### In
- Backend : table SQLite `events` (schema a colonnes typees dediees, aucun champ
  texte libre, aucune IP), creee par `create_all` (zero migration).
- Backend : endpoint `POST /events` anonyme, public, status 201, calque sur
  `POST /feedback`, valide par un modele Pydantic strict (422 hors enums/bornes),
  rate-limite via la fabrique `rate_limiter` de 9.9.
- Backend : event serveur `llm_fallback` persiste depuis `run_full_analysis`
  (app/analysis.py) au point unique du fallback LLM, sans modifier le corps de la
  reponse `/analyze`.
- Backend : logger dedie `events_logger = logging.getLogger("events")` qui ne
  journalise jamais la valeur d'une dimension potentiellement sensible.
- Front : helper `sendEvent(name, props)` dans `frontend/lib/api.ts`, calque sur
  `sendFeedback` (fire-and-forget, jamais awaite, jamais d'exception remontee),
  branche aux handlers reels de `frontend/app/page.tsx` + un composant client
  `<PageViewBeacon>` minimal pour `frontend/app/methode/page.tsx`.
- Tests : `backend/tests/test_events.py` + fixture autouse d'isolation de la table
  `events` dans `backend/tests/conftest.py`.

### Out (non-objectifs, voir §5)
- Tout SaaS analytics externe (GA / Plausible / Umami). On reste first-party.
- Dashboards / rapports d'exploitation des events (lecture, corrélations) : livrable
  distinct ulterieur.
- Cookie / `localStorage` / identifiant de session / fingerprint. Donc les events
  `repeat_session` et `analysis_volume_per_session` cross-visite (corrélation
  inter-visites) sont HORS v1 (reportes v2 conditionnee a un cookie).
- Bandeau / consentement RGPD : non requis (voie sans cookie, exemption mesure
  d'audience). Le wording front « analyses non conservées » reste INCHANGE.
- `feedback_submitted` via `/events` : la donnee est deja en table `feedback` (9.7),
  zero duplication. NE PAS l'ajouter a `/events`.
- Auto-purge / cron de rétention : YAGNI (réveillerait la VM). Purge manuelle
  documentee (voir §3.6).
- Toute modification de `AnalyzeRequest`, `AnalyzeResponse`, du contrat `/analyze`
  ou de la signature `analyzeListing` cote `frontend/lib/api.ts`.
- `prompt_variant` (9.8 différé, dépendance non satisfaite).

## 2. Décisions actées (GATE 1, fermes)

1. **Sans cookie, anonyme, agrege, first-party.** Aucun identifiant, aucun cookie,
   IP non persistée (l'IP ne sert qu'au rate-limit 9.9, jamais stockee ni
   journalisée). Pas de bandeau RGPD. Le wording « analyses non conservées »
   (`frontend/components/design/Footer.tsx:26`, `frontend/app/methode/page.tsx:154`)
   reste inchangé. Les compteurs sont des **proxys de tendance** (fire-and-forget
   best-effort), pas une comptabilité exacte : a acter dans la doc et en commentaire
   « why ».
2. **Schema a colonnes typees dediees** (PAS de props JSON libre, PAS de texte
   libre) : une colonne par dimension connue. Toute valeur hors enum/bool/band
   attendu = rejet 422. C'est le garde-fou anti-PII central.
3. **`llm_fallback` = event serveur** emis au point unique du fallback dans
   `run_full_analysis` (couplé DB), JAMAIS via un flag dans `AnalyzeResponse`. Le
   contrat `/analyze` reste strictement intact.
4. **`page_view` = event « serveur » au sens persiste/compté en base via `POST
   /events`** (voir §2.1 ci-dessous pour la levée d'ambiguïté), declenché au montage
   de `page.tsx` (`useEffect([])`). `methode_view` idem via le beacon client.
5. **`/events` rate-limite LARGE** via `Depends(rate_limiter(limit=120,
   window_seconds=60))` (seuil acté en §3.3). `/feedback` reste a 60/min,
   `/analyze` a 10/min.
6. **Rétention = YAGNI** : pas de cron. Purge manuelle documentee (§3.6).
7. **`feedback_submitted` reste sur `/feedback` seul.** Zéro duplication.

### 2.1 Levée d'ambiguïté : que veut dire « serveur » pour `page_view`
L'analyse 9.10 §3.2 parle de « page_view côté serveur ». Le backend Fly ne sert PAS
les pages (front sur Vercel, SSR Next.js cote Vercel). Le terme « serveur » signifie
ici **« persisté / compté en base via un appel `POST /events` »**, PAS « détecté
sans requête front ». Concrètement :
- `page_view` est emis par un `POST /events` declenché au montage de `page.tsx`
  (`useEffect([])`), donc techniquement un appel front -> backend, mais l'event est
  compté en table `events`.
- `methode_view` est emis par le `POST /events` du composant client
  `<PageViewBeacon>` monté dans le server component `methode/page.tsx`.
- Le SEUL event réellement émis par le code backend lui-même (sans requête front
  dédiée) est `llm_fallback`, inseré au sein du traitement `/analyze`.

Cette distinction doit être citée verbatim dans la SPEC pour que testeur et dev ne
se contredisent pas : un test de `page_view` frappe `POST /events`, il ne « détecte »
pas une visite cote backend.

## 3. Contrat technique

### 3.1 Table `events` (`backend/db/models.py`)
Nouvelle classe `Event(Base)`, `__tablename__ = "events"`, créée automatiquement par
`Base.metadata.create_all` au `init_db()` du startup (comme `feedback`). **AUCUN
`_migrate_events`** (YAGNI, schema complet d'emblée). **AUCUNE colonne texte libre,
aucune IP, aucun identifiant.** Colonnes (toutes les dimensions sont nullable car une
ligne ne porte que les dimensions de son event ; `name` et `created_at` exceptés) :

| Colonne | Type SQLAlchemy | Nullable | Domaine de valeurs | Renseignée par |
|---|---|---|---|---|
| `id` | `Integer` PK autoincrement | non | — | auto |
| `name` | `String` | non | allowlist §3.2 | tous |
| `mode` | `String` | oui | `{url, text}` | `analysis_started` |
| `score_band` | `String` | oui | `{lt40, 40_59, 60_79, 80plus}` | `analysis_succeeded` |
| `confidence` | `String` | oui | `{Élevée, Moyenne, Faible}` | `analysis_succeeded` |
| `pillar_price_status` | `String` | oui | `{aligne, sous, leger_sur, fort_sur, indetermine}` | `analysis_succeeded` |
| `reason` | `String` | oui | `{url_unreachable, no_input, llm_fallback}` | `analysis_failed`, `llm_fallback` |
| `format` | `String` | oui | `{copy, pdf}` | `report_export` |
| `from_scope` | `String` | oui | `{quartier, secteur, ville, metropole, indetermine}` | `district_refine` |
| `to_scope` | `String` | oui | `{quartier, secteur, ville, metropole, indetermine}` | `district_refine` |
| `address_entered` | `Boolean` | oui | `true` / `false` | `address_entered` |
| `path` | `String` | oui | allowlist `{/, /methode}` | `page_view` |
| `referrer_domain` | `String` | oui | hostname seul (domaine), jamais path/query | `page_view` |
| `created_at` | `DateTime` | non | `datetime.utcnow` (defaut serveur) | tous |

Notes de design verrouillées :
- Les libellés d'enum exacts ci-dessus sont contractuels (tests aux valeurs exactes,
  leçon 9.7). Si le dev préfère une autre forme (ex. `lt_40`), il met à jour la SPEC
  et les tests, jamais en silence.
- `confidence` reprend les libellés réels FR de `AnalyzeResponse.confidence`
  (`Élevée|Moyenne|Faible`, cf. `main.py:70`) ; c'est une valeur fermée, non un texte
  libre.
- `pillar_price_status` est dérivée cote front du verdict du pilier prix
  (`res.pillars[0].verdict`) vers l'enum fermée ci-dessus ; le verdict textuel brut
  n'est JAMAIS envoyé.
- Le commentaire de docstring de `Event` rappelle « ni IP, ni identifiant, ni extrait
  d'annonce, ni texte libre » (esprit `Feedback`, models.py:56-57).

### 3.2 Allowlist des noms d'events
`name` est validé contre l'allowlist fermée (toute autre valeur -> 422) :
```
page_view, methode_view, analysis_started, analysis_succeeded,
analysis_failed, report_export, district_refine, address_entered, llm_fallback
```
`feedback_submitted` n'en fait PAS partie (reste sur `/feedback`).
`repeat_session`, `analysis_volume_per_session` n'en font PAS partie (hors v1).

### 3.3 Modèle Pydantic `EventIn` et endpoint `POST /events` (`backend/app/main.py`)
- Nouveau modèle `EventIn(BaseModel)` a cote de `FeedbackIn` (main.py:78). Validation
  **stricte** :
  - `name: str` contraint a l'allowlist §3.2 (`Literal[...]` ou `Enum`) ;
  - chaque dimension est une `Literal`/enum fermée ou un `bool`, jamais un `str`
    libre ; `referrer_domain` et `path` sont les seuls `str` et sont bornés
    (`path` contraint a l'allowlist `{/, /methode}` ; `referrer_domain` est un
    hostname — pas de `/`, pas de `?`, longueur bornée, voir AC11) ;
  - `model_config` interdit les champs inconnus (`extra="forbid"`) -> tout champ non
    déclaré (ex. `url`, `address`, `comment`, `props`) -> 422. C'est le garde-fou
    anti-PII : aucune surface texte libre n'est acceptée.
- Handler `POST /events`, `status_code=201`, calque sur `submit_feedback`
  (main.py:318-345) : ouvre/ferme `SessionLocal()` en `try/finally`, insère un `Event`,
  retourne `{"status": "ok"}`.
- Rate-limit : `_rl: None = Depends(rate_limiter(limit=120, window_seconds=60))`.
  Seuil acté 120/min/IP : large pour ~5-10 events/session légitime, coupe le spam
  d'INSERT.
- Import : ajouter `Event` a `from db.models import Comparable, Feedback`
  (main.py:14 -> `Comparable, Feedback, Event`).
- Logger : `events_logger = logging.getLogger("events")` (a cote de
  `feedback_logger`, main.py:27). Il journalise **au plus** le `name` de l'event et
  des **booléens de présence**, jamais une valeur de dimension potentiellement
  sensible (jamais `referrer_domain`, jamais `path`, jamais une IP). Ajouter `events`
  a la liste des loggers nommés de `backend/CLAUDE.md` §12.
- CORS : aucun changement, `/events` hérite du middleware global (main.py:41-48,
  `allow_origin_regex` vercel + localhost). Appel cross-origin Vercel -> Fly OK.

### 3.4 Event serveur `llm_fallback` (`backend/app/llm_semantic.py` + `app/analysis.py`)
Contrainte ferme : ne PAS coupler `llm_semantic` (cache-only) a la DB, et ne PAS
toucher `AnalyzeResponse`. Mécanisme retenu :
- `_FALLBACK` (llm_semantic.py:114) et/ou le retour de `analyze_semantic`
  (llm_semantic.py:242) porte un marqueur interne **non exposé** dans la réponse HTTP
  (ex. clé `_fallback: True` dans le dict retourné, jamais propagée vers
  `AnalyzeResponse` qui n'a que `global_score, verdict, confidence, pillars, actions,
  local_context` — main.py:67-75). Ce marqueur ne doit JAMAIS apparaître dans le
  corps JSON de `/analyze`.
- `run_full_analysis` (analysis.py:132-136) lit ce marqueur après l'appel
  `analyze_semantic` et, si le fallback a été servi, insère un `Event(name="llm_fallback",
  reason="llm_fallback")` en base (`SessionLocal()`), puis poursuit le traitement
  normal. L'insertion est best-effort : une erreur DB ne doit pas faire échouer
  `/analyze` (capturer/logguer sans relever).
- Conséquence : `analysis_failed(reason=llm_fallback)` n'est JAMAIS émis cote front
  (le front voit un 200 normal). Le front n'émet que `analysis_failed` avec
  `reason ∈ {url_unreachable, no_input}`.

### 3.5 Émission front (`frontend/lib/api.ts`, `frontend/app/page.tsx`, `methode/page.tsx`)
`sendEvent(name: string, props?: object): void` (ou `Promise<void>` non awaité),
calqué sur `sendFeedback` (api.ts:96-111) : `fetch(NEXT_PUBLIC_API_URL + "/events",
{method:"POST", headers, body})` dans un `try/catch` qui avale toute erreur, **jamais
awaité dans un chemin critique**, jamais d'exception remontée. `keepalive: true`
optionnel (ceinture bon marché). Points d'accroche (handlers réels vérifiés) :

| Event | Point d'accroche (vérifié) | Props envoyées | Interdit (PII) |
|---|---|---|---|
| `page_view` | nouveau `useEffect([])` de montage dans `page.tsx` | `path`, `referrer_domain` | jamais path/query du referrer |
| `methode_view` | `<PageViewBeacon name="methode_view">` (client) monté dans `methode/page.tsx` (server component) | aucune | — |
| `analysis_started` | début de `handleAnalyze` (page.tsx:485) | `mode ∈ {url,text}` | jamais `value`/URL/texte |
| `analysis_succeeded` | après `setResult(res)` (page.tsx:493-496) | `score_band`, `confidence`, `pillar_price_status` | jamais le score exact en clair via une autre dimension, jamais le verdict textuel |
| `analysis_failed` | `catch` de `handleAnalyze` (page.tsx:497-501) | `reason ∈ {url_unreachable, no_input}` | jamais le message d'erreur brut |
| `report_export` | `handleCopy` (page.tsx:596 -> `copy`), `handlePrint` (page.tsx:607 -> `pdf`) | `format ∈ {copy,pdf}` | — |
| `district_refine` | `handleRefine` (page.tsx:506) | `from_scope`, `to_scope` (avant/après) | — |
| `address_entered` | `handleAddressSubmit` (page.tsx:528) | `address_entered: boolean` | jamais `addressInput` |

Garde-fous front fermes :
- `sendEvent` n'est JAMAIS awaité dans `handleAnalyze` / `handleRefine` : l'event ne
  doit ni retarder ni faire échouer l'analyse. Le `try/catch` de `analyzeListing` ne
  voit QUE l'erreur d'analyse.
- `analysis_started` est émis UNIQUEMENT dans `handleAnalyze`, jamais dans
  `handleRefine` (sinon `taux_lancement` faussé par les refines). `district_refine`
  est son propre event.
- `referrer_domain` = `new URL(document.referrer).hostname` (domaine seul) ; jamais
  `document.referrer` brut. Referrer absent -> dimension omise.
- `<PageViewBeacon>` est un composant `"use client"` minimal (`useEffect([])` +
  `sendEvent`). NE PAS passer `methode/page.tsx` en client (casserait l'export
  `metadata`, methode/page.tsx:6-10, et le SSR/SEO).
- **Garde double-comptage StrictMode** : `page_view` et `methode_view` utilisent un
  `useRef` « déjà émis » pour idempotence intra-montage (React StrictMode monte deux
  fois en dev ; sans garde, le compteur de dev est faux). Acté.

### 3.6 Rétention / purge (YAGNI + manuelle documentée)
Pas de cron. Option retenue (la plus simple) : **purge manuelle documentée**, sans
nouvel endpoint. Documenter dans `backend/CLAUDE.md` §9 (ou voisin) une commande
`DELETE FROM events WHERE created_at < :date` lançable en SSH sur la VM Fly
(`/data/comparables.db`), avec la politique : « events conservés tant que le volume le
permet ; purge manuelle disponible ». Aucun `X-Admin-Token` ni endpoint ajouté (le
volume FORECAST — quelques k lignes/mois — ne le justifie pas). Si un endpoint admin
devient nécessaire plus tard, réutiliser le pattern `POST /admin/comparables/maintenance`.

### 3.7 Schéma front / contrat API
- `frontend/lib/api.ts` : AJOUT de `sendEvent` uniquement. `analyzeListing` et
  `sendFeedback` inchangés. `AnalyzeResponse`/`ApiResult` inchangés.
- Aucun nouveau secret, variable d'env, ou dépendance Python (réutilise `rate_limiter`,
  `requirements.txt` intact).

### 3.8 Isolation des tests (leçon 9.7 / 9.9)
La table `events` est un **état partagé persistant** (fichier SQLite jetable de la
suite). Garde-fou requis, au même titre que le reset rate-limit :
- `backend/tests/conftest.py` : fixture **autouse, scope function** qui vide la table
  `events` AVANT chaque test (`DELETE FROM events` / `Event.__table__.delete()`),
  import protégé. Ne jamais dépendre de l'ordre des tests ni d'un `count()` global
  inter-tests.
- Les assertions de persistance filtrent sur les lignes du test courant (état vide en
  entrée garanti), jamais sur un `count()` absolu accumulé.
- Les bornes d'enum sont testées aux valeurs EXACTES (valides ET invalides), leçon 9.7.

## 4. Critères d'acceptation (testables, pytest / revue front)

Fichier backend cible : `backend/tests/test_events.py`. Valeurs exactes a tester.
Numérotés pour mapping 1:1 testeur.

**AC1 — Table `events` créée par `create_all`, sans migration.** Après `init_db()`,
la table `events` existe (ex. `inspect(engine).has_table("events")` ou un INSERT/SELECT
réussit). Aucun appel a un `_migrate_events` n'est requis ; `backend/db/session.py`
n'introduit pas de fonction `_migrate_events` (assertion statique : `_migrate_events`
absent du module).

**AC2 — `POST /events` 201 sur payload valide de chaque event.** Pour chacun des 9
noms de l'allowlist §3.2 émissibles cote client (`page_view`, `methode_view`,
`analysis_started`, `analysis_succeeded`, `analysis_failed`, `report_export`,
`district_refine`, `address_entered` ; `llm_fallback` testé via AC8/serveur), un
payload minimal valide renvoie `status_code == 201` et `{"status": "ok"}`, et une
ligne `events` correspondante est persistée avec le bon `name`.

**AC3 — 422 sur `name` hors allowlist.** `POST /events {"name": "haxor"}` (ou tout
nom absent de §3.2, ex. `feedback_submitted`, `repeat_session`) renvoie
`status_code == 422`. Aucune ligne n'est insérée.

**AC4 — 422 sur valeur d'enum invalide.** Chacun de ces payloads renvoie
`status_code == 422` : `report_export` avec `format="xml"` ;
`analysis_failed` avec `reason="boom"` ; `analysis_started` avec `mode="voice"` ;
`district_refine` avec `to_scope="galaxie"` ; `analysis_succeeded` avec
`score_band="42"`. Aucune ligne insérée.

**AC5 — 422 / refus de tout champ texte libre ou PII.** `POST /events` avec un champ
non déclaré (ex. `{"name":"analysis_started","mode":"url","url":"https://x.fr/annonce"}`,
ou un champ `address`, `comment`, `props`, `raw_text`, `ip`) renvoie
`status_code == 422` (via `extra="forbid"`). Vérifie qu'aucune surface texte libre
n'entre dans `events`.

**AC6 — `address_entered` est un booléen, jamais une chaîne d'adresse.** `POST /events
{"name":"address_entered","address_entered":true}` -> 201. Un payload qui tenterait
d'envoyer une adresse en clair (`{"name":"address_entered","address_entered":"3 rue X"}`)
renvoie 422 (type bool strict), OU le champ adresse libre est rejeté par
`extra="forbid"`. La colonne stockée est bien un booléen.

**AC7 — Rate-limit `/events` actif (429 au-delà du seuil).** Avec une IP `Fly-Client-IP`
fixe, les 120 premières requêtes `/events` valides ne renvoient pas 429 ; la 121e
renvoie `status_code == 429` avec un en-tête `Retry-After` entier `>= 1` et `<= 60`.
Une IP `Fly-Client-IP` différente n'est pas bloquée (buckets distincts). La fixture
autouse de reset rate-limit (conftest, leçon 9.9) garantit l'isolation.

**AC8 — `llm_fallback` persisté quand le fallback est servi, SANS changer `/analyze`.**
En forçant `analyze_semantic` a retourner `_FALLBACK` (échec LLM simulé), un appel
`run_full_analysis` (ou `POST /analyze`) :
(a) persiste exactement une ligne `events` avec `name="llm_fallback"` et
`reason="llm_fallback"` ;
(b) le corps de la réponse `/analyze` reste strictement le schéma attendu
(`global_score, verdict, confidence, pillars, actions`, `local_context` optionnel) et
ne contient AUCUNE clé `_fallback` / `degraded` / marqueur interne (assertion :
`set(response.json().keys()) <= {global_score, verdict, confidence, pillars, actions,
local_context}`).
Inversement, un appel sans fallback (LLM nominal) n'insère AUCUNE ligne `llm_fallback`.

**AC9 — Le logger `events` ne fuit aucune dimension sensible.** Avec `caplog` au niveau
DEBUG, après un `POST /events` portant `referrer_domain="exemple.fr"`,
`path="/methode"` et un `POST /events` portant des dimensions enum, `caplog.text` ne
contient ni `"exemple.fr"`, ni la valeur d'une dimension fermée potentiellement
sensible, ni aucune IP. Le logger peut contenir le `name` et des booléens de présence.

**AC10 — `referrer_domain` borné au domaine (jamais path/query).** `POST /events` avec
`{"name":"page_view","path":"/","referrer_domain":"google.com/search?q=x"}` (valeur
contenant `/` ou `?`) renvoie `status_code == 422` (un hostname ne contient ni `/`
ni `?`). `referrer_domain="google.com"` -> 201. (Le tronquage au hostname est fait
cote front via `URL(...).hostname` ; le backend refuse une valeur non-hostname comme
garde-fou serveur.)

**AC11 — `path` borné a l'allowlist interne.** `POST /events`
`{"name":"page_view","path":"/methode"}` -> 201 ; `{"name":"page_view","path":"/"}`
-> 201 ; `{"name":"page_view","path":"/admin"}` ou un path d'URL d'annonce
(`"https://x.fr/bien"`) -> `status_code == 422`. Aucun path inattendu n'entre en base.

**AC12 — Aucune colonne IP / identifiant dans `events`.** Test statique sur le modèle :
les colonnes de `Event.__table__.columns` sont exactement l'ensemble de §3.1 et ne
contiennent ni `ip`, ni `user_id`, ni `session_id`, ni `props`, ni aucune colonne de
type texte libre non bornée hors `referrer_domain`/`path` (qui sont bornées par AC10/AC11).

**AC13 — Aucune nouvelle dépendance.** `backend/requirements.txt` reste l'ensemble de
9.9 (`{fastapi, uvicorn[standard], sqlalchemy, requests, beautifulsoup4, openai,
numpy}`) et garde 7 lignes non vides. `/events` réutilise `rate_limiter`.

**AC14 — Isolation `events` entre tests effective.** Deux tests consécutifs qui
insèrent chacun via `/events` voient un état initial vide (la fixture autouse de §3.8
vide la table avant chaque test). Falsifiable : un test qui insère N lignes n'impacte
pas le `count()` d'un test indépendant suivant qui repart de 0.

**AC15 — Contrat `/analyze` et `/feedback` inchangés.** `POST /analyze {}` renvoie
toujours `400` (aucun input), `POST /analyze` valide renvoie le schéma habituel ;
`POST /feedback {"rating":3}` renvoie toujours `201 {"status":"ok"}` et reste a 60/min.
L'introduction de `/events` ne modifie aucun code métier ni rate-limit existant.

**Critères front (vérification par revue, non pytest backend) :**

**AC16 — `sendEvent` fire-and-forget, calque `sendFeedback`.** Revue de
`frontend/lib/api.ts` : `sendEvent` existe, poste sur `/events`, enveloppe le `fetch`
