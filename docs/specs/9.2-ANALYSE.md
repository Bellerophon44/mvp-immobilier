## 1. Objectif et perimetre

### 1.1 Reformulation
A la fin d'un `POST /analyze` reussi (HTTP 200), si l'utilisateur a fourni un
email **et** coche une case de consentement explicite, lui envoyer par email le
recapitulatif de l'analyse deja calculee : score global, verdict, confiance, les
3 piliers (prix / transparence / risques) avec verdict + explication, et les 2
listes d'actions (`questions`, `negotiation`). Provider suggere : Resend (free
tier 100 mails/jour), secret `RESEND_API_KEY`, domaine expediteur verifie.

Remarque importante : le requirement parle de "3 listes d'actions" mais le code
reel n'en a plus que **2** depuis le chantier A (`actions` = `{questions,
negotiation}`, `to_check` fusionne dans `questions`, cf. `backend/CLAUDE.md` §11bis
A et `app/analysis.py:181-184`). Le mail doit refleter ces 2 listes, plus le bloc
`local_context` non-score si on veut etre complet (optionnel).

### 1.2 Dans le perimetre (IN)
- Recevoir un email + un consentement booleen cote backend.
- Composer un corps de mail (texte et/ou HTML) a partir du dict deja produit par
  `run_full_analysis` (aucun re-calcul, aucun nouvel appel LLM).
- Declencher l'envoi via un provider transactionnel.
- Garde-fous anti-abus et conformite RGPD minimale.
- Champ email + checkbox de consentement cote frontend.

### 1.3 Hors perimetre (OUT) — a confirmer
- Pas de compte utilisateur / authentification (le requirement 9.6 en a besoin,
  pas 9.2).
- Pas de double opt-in (lien de confirmation par mail) sauf decision contraire.
- Pas d'historique des analyses envoyees consultable par l'utilisateur.
- Pas de desinscription / gestion de preferences (il n'y a pas d'envoi recurrent
  ici : un mail = une action ponctuelle declenchee par l'utilisateur lui-meme).
- Pas de PDF en piece jointe (HTML inline suffit pour un MVP).

## 3. Dependances et ordre

### 3.1 Prerequis EXTERNES (bloquants, action humaine seule)
Ces trois points ne peuvent pas etre faits par un agent ; ils conditionnent la
mise en prod (mais pas le developpement, qui peut etre mocke) :
1. **Creer un compte chez le provider** (Resend ou autre).
2. **Verifier un domaine expediteur** (enregistrements DNS SPF/DKIM). Sans domaine
   verifie, soit l'envoi est refuse, soit les mails partent en spam. Le projet n'a
   **pas de domaine custom acquis** (CONTEXT §3.2 : "Domaine custom : non acquis").
   C'est un point dur : `noreply@mvp-immobilier.fr` n'existe pas encore. Resend
   propose un domaine de test `onboarding@resend.dev` limite aux mails vers sa
   propre adresse de compte — inutilisable pour de vrais utilisateurs.
3. **Poser le secret** `RESEND_API_KEY` via `fly secrets set ... --app
   backend-frosty-sound-441-docker`.

Consequence : **9.2 ne peut pas etre livre en prod sans qu'un domaine soit acquis
et verifie.** C'est un prerequis plus lourd que ne le laisse penser l'estimation
"2-3h" du requirement. Le code peut etre ecrit et teste avant, derriere un
garde-fou 503 (feature inactive tant que le secret manque), comme `/admin/comparables`.

### 3.2 Dependances internes
- Depend de la stabilite du schema de sortie `run_full_analysis` (acquis).
- Aucune dependance vers 9.6 (watchlist) ni 9.7 (feedback) : 9.2 est autonome.
- 9.7 fournit un **modele reutilisable** (table, endpoint POST 201, composant de
  formulaire, mention RGPD), ce qui reduit l'effort reel.

### 3.3 Prerequis manquant signale
- Domaine expediteur : **manquant** (cf. §3.1.2). Bloquant prod.
- Pas de limite de taux sur `/analyze` (CONTEXT §8 dette : "Rate limiting sur
  /analyze (anti-abus)" a faire). 9.2 cree un nouveau vecteur d'abus (envoi de
  mail) alors qu'aucun rate limit n'existe. Voir §4.

## 4. Risques d'anti-pattern et points de vigilance

1. **Nouveau vendor + nouvelle dependance** (anti-pattern implicite : le projet
   est volontairement minimal). Resend = un service externe de plus, un secret de
   plus, un point de panne de plus. A peser contre la valeur (engagement/retention,
   non valide — CONTEXT §2.4 hypotheses non validees). Challenge en §6.

2. **Anti-abus / spam (le plus serieux).** Un endpoint qui envoie un email a une
   adresse fournie dans la requete est un **relais de spam ouvert** : un attaquant
   peut envoyer des mails a des tiers arbitraires (harcelement, reputation du
   domaine cramee, free tier 100/jour epuise, blacklistage du domaine fraichement
   verifie). Il n'y a **aucun rate limit** aujourd'hui (CONTEXT §8). Garde-fous
   minimaux a exiger : limite de debit (par IP et/ou global/jour), validation
   stricte du format email, et idealement le mail ne contient QUE l'analyse que
   l'utilisateur vient de declencher (pas de contenu arbitraire injectable). Le
   risque de griller la reputation d'un domaine tout neuf est concret.

3. **RGPD — email = donnee personnelle.** Exigences : consentement **explicite**
   (case NON pre-cochee, lessons.md "consentement explicite avant stockage"),
   finalite claire ("recevoir cette analyse par mail"), minimisation, mention
   "vos donnees ne sont pas reutilisees" (CONTEXT §9.2). Question ouverte : faut-il
   **stocker** l'email ? La table `Feedback` (`db/models.py:54-58`) a justement ete
   concue en minimisation ("ni IP, ni identifiant"). Stocker un email serait un pas
   en arriere sur la posture RGPD du projet. Voir Option D.

4. **Fallback LLM envoye par mail.** Si OpenAI echoue, `analyze_semantic` renvoie
   `_FALLBACK` (`llm_semantic.py:114-121` : "Analyse indisponible.", listes vides)
   et `/analyze` repond quand meme 200. Envoyer ce contenu vide par mail degrade
   l'image produit. Garde-fou a prevoir : ne pas envoyer si le resultat est un
   fallback (detectable : `summary == "Analyse indisponible."` ou un flag a
   remonter depuis `analyze_semantic`).

5. **Redistribution d'annonce brute (anti-pattern CONTEXT §11.3).** A confirmer :
   le mail ne doit contenir QUE les agregats/score/verdict/actions, jamais le texte
   brut de l'annonce ni des comparables individuels. Le dict de
   `run_full_analysis` ne contient deja pas de contenu brut d'annonce ni de
   comparable nominatif (uniquement mediane/quartiles agreges via les explications
   des piliers). Donc OK par construction, a verifier au moment d'ecrire le
   template (ne pas re-injecter `raw_text`).

6. **Cout.** Free tier Resend = 100 mails/jour, 3000/mois. A l'echelle MVP (<1
   euro/mois, trafic faible) : nul. Risque seulement en cas d'abus (cf. point 2),
   d'ou l'importance du rate limit.

7. **Template HTML : nouvelle dependance Jinja2 ?** Le projet n'utilise pas Jinja2.
   Pour un mail unique, un template HTML inline (f-string Python avec echappement)
   suffit et evite une dependance. Jinja2 = sur-dimensionne pour 1 template. Voir
   Option F. Attention echappement HTML du contenu LLM (XSS dans le client mail /
   injection) : echapper les chaines inserees.

8. **Rupture de contrat API.** L'ajout de champs optionnels a `AnalyzeRequest` ne
   rompt pas le contrat serveur, mais la regle du repo impose la MAJ de
   `lib/api.ts` (anti-pattern CONTEXT §11.9, lessons.md). A respecter.

9. **Couplage envoi/latence de /analyze.** `/analyze` est deja lent (appel LLM +
   eventuel fetch URL + reveil machine Fly auto-stop). Ajouter un appel HTTP
   synchrone vers Resong allonge la reponse et la rend dependante de la dispo de
   Resend. Voir Option C (sync vs async).

## 5. OPTIONS chiffrees (choix structurants a arbitrer)

### Option A vs B — Ou recevoir l'email (contrat API)

**Option A : ajouter `email` + `consent` a `AnalyzeRequest`** (suggere par le
requirement, etape 1).
- Avantages : un seul appel ; le contenu est deja en main cote serveur (zero
  re-calcul, zero appel LLM supplementaire) ; simple.
- Inconvenients : melange deux responsabilites (analyser / envoyer un mail) dans
  une route deja chargee ; impose la MAJ de `lib/api.ts` (regle repo) ; on demande
  l'email **avant** que l'utilisateur ait vu la valeur de l'analyse (mauvais
  moment de conversion — l'utilisateur coche avant de savoir si ca vaut le coup).

**Option B : endpoint separe `POST /send-report`.**
- Variante B1 (recommandee techniquement) : le front renvoie le **payload deja
  recu** (le dict `ApiResult` complet) + email + consent. Le backend ne fait que
  composer et envoyer. Cout LLM : **zero** (pas de re-analyse). Risque : le client
  envoie le contenu du mail -> verifier qu'on ne devient pas un relais d'envoi de
  contenu arbitraire (cf. §4.2) ; mitiger par rate limit + le fait que le contenu
  reste un recap d'analyse (pas un champ libre vers un tiers).
- Variante B2 (a eviter) : `/send-report` ne recoit que `raw_text`/`url` + email
  et **re-declenche** `run_full_analysis` -> **un appel LLM en plus** (~0,001 euro,
  surtout : 2x la latence et le risque fallback). Inutile puisque le front a deja
  le resultat.
- Avantages B : separation des responsabilites ; `/analyze` intouche (pas de
  rupture de contrat, pas de MAJ forcee de `lib/api.ts` au sens d'un changement de
  signature de /analyze) ; l'email est demande APRES affichage du resultat (bon
  moment) ; rate limit cible sur l'endpoint d'envoi sans penaliser l'analyse.
- Inconvenients B : un endpoint de plus a securiser ; en B1 le contenu transite
  client->serveur (taille + confiance).

**Recommandation analyste : Option B1.** Elle evite la rupture de contrat
`/analyze`, place la demande d'email au bon moment (apres resultat, comme le
feedback 9.7), et n'engendre aucun cout LLM. Le pattern existe deja : `/feedback`
+ `sendFeedback` sont exactement ce modele (POST separe, non bloquant, le front
renvoie `global_score`/`verdict` deja connus). A arbitrer (Q1).

### Option C — Envoi synchrone vs asynchrone
- **C1 Synchrone** dans la requete : simple, mais la reponse depend de Resend
  (latence + dispo). Sur Option B1, acceptable car `/send-report` ne fait que
  l'envoi (pas l'analyse) ; l'utilisateur attend une confirmation d'envoi.
- **C2 BackgroundTasks FastAPI** : repond tout de suite, envoie ensuite.
  **Contrainte Fly auto-stop** (CONTEXT §0, fly.toml `auto_stop_machines=true`,
  `min_machines_running=0`) : si la machine s'arrete juste apres avoir repondu, une
  background task non terminee peut etre coupee. Pour un envoi de quelques
  centaines de ms, le risque est faible mais reel (pas de garantie de livraison).
- **C3 File d'attente** (Redis/worker) : sur-dimensionne pour un MVP <1 euro/mois.
  A exclure.

**Recommandation : C1 synchrone si Option B1** (l'utilisateur veut une confirmation
"mail envoye", et l'endpoint ne fait que ca) ; **C2 BackgroundTasks si Option A**
(pour ne pas allonger `/analyze`). A arbitrer avec Q1/Q5.

### Option D — Persistance de l'email (RGPD)
- **D1 Ne rien stocker** : envoyer puis oublier. Minimisation maximale, coherent
  avec la posture de la table `Feedback` (zero donnee perso). Pas de table, pas de
  retention a gerer.
- **D2 Stocker email + consent + horodatage** (preuve de consentement) : utile si
  on veut tracer le consentement RGPD, mais cree un fichier de donnees perso a
  proteger, a purger, a documenter (registre de traitement, retention).
- **D3 Stocker un hash de l'email** (anti-doublon / anti-abus) sans l'email en
  clair : compromis, mais peu utile sans cas d'usage (pas d'envoi recurrent).

**Recommandation : D1 (ne rien stocker).** Pour un envoi ponctuel a l'initiative
de l'utilisateur, la minimisation prime. La preuve de consentement par un log
non-nominatif (timestamp + flag, sans email) peut suffire. A arbitrer (Q3).

### Option E — SDK Resend vs requests vs autre provider
- **E1 SDK `resend`** : ajoute 1 dependance, ergonomique.
- **E2 `requests`** (deja present, requirements.txt:4) sur l'API REST Resend :
  zero nouvelle dependance, ~20 lignes. Coherent avec le minimalisme du projet.
- Provider : Resend (free 100/j) vs Postmark (100/mois gratuit, tres bonne
  delivrabilite transactionnelle) vs Brevo (300/j gratuit) vs SMTP perso (gratuit
  mais delivrabilite/maintenance penibles, a eviter). Tous imposent un domaine
  verifie pour un usage serieux.

**Recommandation : Resend via `requests` (E2)** pour eviter une dependance, sauf
preference humaine pour un autre provider. A arbitrer (Q2 + Q6 action externe).

### Option F — Template HTML inline vs Jinja2
- **F1 Inline** (f-string + helper d'echappement HTML) : zero dependance, suffisant
  pour 1 template. Reference de structure deja ecrite : `handleCopy`
  (`page.tsx:372-410`).
- **F2 Jinja2** : propre pour beaucoup de templates, mais sur-dimensionne ici (1
  seul mail). Nouvelle dependance.
- Penser a un fallback **texte brut** (multipart) pour la delivrabilite.

**Recommandation : F1 inline (HTML + texte), pas de Jinja2.** A arbitrer (Q7).

## 6. Challenge du requirement lui-meme (posture adversariale)

- **Valeur non validee.** L'objectif (engagement/retention) repose sur des
  hypotheses d'acquisition explicitement "non validees" (CONTEXT §2.4). Avant de
  brancher un vendor, un secret et un domaine, l'alternative **"Copier le rapport"
  existe deja** (`handleCopy`, page.tsx:372) et couvre 80% du besoin "garder une
  trace" a cout nul et risque nul. Une option "telecharger en .txt/.md" cote front
  est encore plus simple et evite TOUT le sujet RGPD/spam/domaine.
- **Sur-cout cache.** L'estimation "2-3h" ignore le prerequis domaine (acquisition
  + verification DNS), l'anti-abus (rate limit absent), et les tests RGPD. Le cout
  reel est superieur, surtout la partie non-codable (humaine).
- **Moment du MVP.** Pour un MVP <1 euro/mois en phase de test utilisateurs, ouvrir
  un vecteur d'envoi de mail (donc d'abus et de reputation de domaine) est un pari
  asymetrique : peu de gain prouve, risque operationnel reel.
- **Proposition de degrade.** Si la valeur "garder l'analyse" est le vrai besoin :
  livrer d'abord un export client (.md/.txt ou impression PDF navigateur), zero
  backend, zero RGPD, zero vendor. Garder 9.2 (vrai email) pour quand un domaine
  sera acquis et qu'un rate limit existera. Ceci est remonte en Q0.

## 7. QUESTIONS POUR L'HUMAIN (GATE 1)

> Distinction : [ACTION EXTERNE] = l'humain doit faire quelque chose hors code
> (compte, DNS, secret), bloquant pour la prod. [ARBITRAGE] = decision de
> conception, implementable ensuite par un agent.

**Q0 — [ARBITRAGE, prealable] Faut-il vraiment faire 9.2 maintenant, ou un export
client suffit-il ?**
Le bouton "Copier le rapport" existe deja. Un export .md/.txt ou impression PDF
cote navigateur couvre "garder l'analyse" sans vendor, sans RGPD, sans domaine.
- Reco : livrer l'export client d'abord ; reporter le vrai email tant qu'aucun
  domaine n'est acquis et qu'aucun rate limit n'existe. Si l'humain veut l'email
  malgre tout, passer aux questions suivantes.

**Q1 — [ARBITRAGE] Contrat : champ email dans `/analyze` (Option A) ou endpoint
separe `POST /send-report` recevant le resultat deja calcule (Option B1) ?**
- Reco : **B1**. Pas de rupture de contrat `/analyze`, demande de l'email apres
  affichage du resultat (meilleur moment), zero cout LLM, reutilise exactement le
  pattern `/feedback` + `sendFeedback`.

**Q2 — [ARBITRAGE] Provider et integration : Resend via `requests` (zero nouvelle
dependance) ou via le SDK `resend`, ou autre provider (Postmark/Brevo) ?**
- Reco : **Resend via `requests`** (Option E2). Si delivrabilite transactionnelle
  prioritaire, Postmark est un bon choix mais avec son propre compte/domaine.

**Q3 — [ARBITRAGE] Persistance : ne rien stocker (D1), stocker email+consent+date
(D2), ou hash de l'email (D3) ?**
- Reco : **D1, ne rien stocker.** Coherent avec la posture minimisation de la table
  `Feedback`. Tracer le consentement par un log non-nominatif (timestamp + flag).

**Q4 — [ARBITRAGE] Emplacement UI du champ email + checkbox : dans la zone resultat
(comme `FeedbackForm`, apres affichage) ou dans `AnalyzerInput` (avant analyse) ?**
- Reco : **zone resultat**, sous le resultat, case de consentement NON pre-cochee,
  bouton desactive tant que email valide + case cochee ne sont pas reunis (modele
  `FeedbackForm`).

**Q5 — [ARBITRAGE] Envoi synchrone (confirmation "mail envoye") ou asynchrone
(BackgroundTasks) compte tenu de l'auto-stop Fly ?**
- Reco : **synchrone** si Option B1 (endpoint dedie, l'utilisateur attend une
  confirmation). Eviter la file d'attente (sur-dimensionne).

**Q6 — [ACTION EXTERNE, BLOQUANT PROD] Acquerir et verifier un domaine expediteur,
creer le compte provider, poser le secret.**
Trois actions que seul l'humain peut faire, indispensables pour envoyer a de vrais
utilisateurs (le domaine de test Resend n'envoie qu'a soi-meme) :
1. Acquerir un domaine (le projet n'en a pas — CONTEXT §3.2).
2. Creer le compte Resend (ou autre) et verifier le domaine (DNS SPF/DKIM).
3. `fly secrets set RESEND_API_KEY=... --app backend-frosty-sound-441-docker`.
- Reco : tant que ce n'est pas fait, livrer le code derriere un garde-fou 503
  (feature inactive si secret absent, modele `_check_admin_token`) afin de
  developper/tester sans bloquer.

**Q7 — [ARBITRAGE] Template : HTML inline + texte brut (F1) ou Jinja2 (F2) ?**
- Reco : **F1 inline** (HTML + fallback texte), avec echappement des chaines LLM ;
  pas de nouvelle dependance pour un unique template.

**Q8 — [ARBITRAGE] Anti-abus : quel garde-fou minimal contre l'usage en relais de
spam (endpoint qui envoie a une adresse fournie) ?**
- Reco : rate limit (par IP + plafond global/jour aligne sur le free tier),
  validation stricte du format email, et le mail ne contient que le recap
  d'analyse (pas de champ libre vers un tiers). Decider si un double opt-in est
  necessaire (reco : non pour un MVP, mais ne pas stocker l'email compense).

**Q9 — [ARBITRAGE] Que faire si l'analyse est un fallback LLM (contenu vide) ?**
- Reco : ne pas envoyer de mail dans ce cas ; remonter un flag de fallback depuis
  `analyze_semantic` ou detecter `summary == "Analyse indisponible."`. A confirmer.
