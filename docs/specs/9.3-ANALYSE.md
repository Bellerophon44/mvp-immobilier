## 1. Objectif et périmètre

### 1.1 Reformulation de l'objectif
Détecter automatiquement que l'IA d'analyse renvoie des réponses de repli
(`_FALLBACK`), signe que l'appel OpenAI échoue, et alerter un humain quand cela
dépasse un seuil — sans exposer de contenu d'annonce ni de secret, à coût
quasi nul (MVP < 1 €/mois).

### 1.2 Le mécanisme réel à surveiller (ancrage code)
Un seul point de génération du fallback existe, contrairement à ce que suggère
le requirement (« log warning "LLM call failed" OU retour `_FALLBACK` » — ce ne
sont pas deux signaux distincts mais le même événement) :

- `backend/app/llm_semantic.py:223-242` — `analyze_semantic()` enveloppe l'appel
  `client.chat.completions.create(...)` et `json.loads(content)` dans un
  `try/except Exception`.
- `backend/app/llm_semantic.py:236-241` — sur exception :
  `logger.exception("LLM call failed (model=%s, key_set=%s): %s", ...)`.
  Logger nommé `llm_semantic` (déclaré `:10`). Le message exact loggé est
  `LLM call failed` au niveau ERROR (logger.exception => ERROR + traceback).
- `backend/app/llm_semantic.py:242` — `return dict(_FALLBACK)` (constante
  `_FALLBACK` définie `:114-140`).

Conséquence importante pour la détection : le fallback est **uniquement** émis
par `analyze_semantic`. Ce qui déclenche le fallback couvre deux familles :
1. échec réseau / API OpenAI (timeout, 429, 5xx, clé invalide, quota dépassé) ;
2. réponse non-JSON parsable (`json.loads` échoue).
Les deux passent par le même `except` et le même log. La string `_FALLBACK`
n'apparaît jamais dans les logs : c'est un nom de variable, pas un texte loggé.
Le seul marqueur loggable fiable est donc **`LLM call failed`** (logger
`llm_semantic`, niveau ERROR).

Note de robustesse : le cache (`:216-219`) peut renvoyer un fallback déjà mis en
cache ? Non — `_set_cache` n'est appelé que `:275`, après le succès ; le
fallback du `except` n'est jamais mis en cache. Bon point : un échec ponctuel ne
« colle » pas. Chaque fallback correspond donc bien à un appel OpenAI raté
distinct, jamais à un re-service de cache.

### 1.3 In / Out

In (cœur du requirement) :
- Compter / observer les occurrences de fallback LLM.
- Décider d'un seuil au-delà duquel on considère qu'il y a dégradation.
- Notifier un humain.

Out (à exclure explicitement du périmètre 9.3, sauf arbitrage contraire) :
- Monitoring d'erreurs généraliste (Sentry) — c'est une autre ligne de dette
  (CONTEXT §4.3, §8, CLAUDE §11).
- Surveillance des autres `logger.warning` (fetch URL, scrapers, géocodage) :
  ce sont des dégradations de collecte/fetch, pas de la qualité IA. Les
  englober élargirait le périmètre sans justification.
- Alertes de coût OpenAI : couvert par 9.4 (config UI sur platform.openai.com,
  pas de code). Ne pas mélanger.
- Évaluation qualitative du contenu des réponses LLM (pertinence des
  questions/leviers) : c'est de l'A/B testing / feedback (9.7 livré, 9.8). Le
  9.3 ne juge pas la qualité sémantique, seulement la disponibilité.

## 3. Dépendances et ordre

- 9.3 ne dépend d'aucun autre requirement pour la partie détection. Le
  mécanisme `_FALLBACK` est en place et stable.
- Le **canal email** dépendrait d'un domaine + provider : c'est exactement la
  dépendance qui a bloqué 9.2 (CONTEXT §9.2 : « Domaine vérifié pour
  expéditeur »). Prérequis email **manquant** aujourd'hui. Donc, si
  notification email retenue, 9.3 hérite du blocage de 9.2 → à ne pas choisir
  pour un livrable autonome.
- Le **canal Slack** ne dépend que d'un webhook entrant (URL secrète), pas de
  domaine : prérequis léger, action externe simple.
- L'option polling par cron GitHub Actions réutilise le pattern de
  `collect.yml` : pas de nouvelle dépendance d'outillage CI.
- Aucun lien de précédence avec 9.4 (coût OpenAI), 9.7 (feedback, déjà livré),
  9.8 (A/B prompts). 9.3 est indépendant.

## 4. Risques d'anti-pattern

1. **Fuite de contenu d'annonce / secret par l'endpoint de santé** (CLAUDE §1,
   §3). Un `GET /admin/health/llm` ne doit renvoyer que des agrégats
   (compteur, fenêtre temporelle, dernier timestamp d'échec) — jamais le texte
   d'annonce, ni le message d'exception OpenAI (qui peut contenir des fragments
   de requête ou d'identifiants). Doit être protégé par `X-Admin-Token` comme
   les autres endpoints admin (`main.py:119-125`). Risque RGPD si on logge le
   contenu : déjà évité dans le code existant (`feedback` ne logge pas le
   commentaire, `main.py:325-326` ; l'analyze ne logge qu'un preview 60 chars,
   `:262`). Rester sur cette ligne.
2. **Nouveau vendor (Better Stack / Logtail)** = dépendance externe + surface de
   coût + un compte de plus à gérer pour un MVP solo. Contraire à l'esprit
   « < 1 €/mois, dépendances minimales ». À challenger fortement (voir §6 et §7).
3. **Log drain Fly** = configuration infra hors-repo, non versionnée, fragile
   (se perd si l'app est recréée), et qui suppose un endpoint HTTPS récepteur
   (donc soit un vendor, soit un service à héberger). Sur-ingénierie probable
   pour le besoin.
4. **Compteur en mémoire volatile + auto-stop Fly** : `fly.toml:12-14`
   (`auto_stop_machines = true`, `min_machines_running = 0`). À très faible
   trafic, la VM s'arrête dès qu'elle est idle → tout compteur en mémoire est
   **remis à zéro** à chaque réveil. Un « taux de fallback/heure » calculé en
   mémoire est donc structurellement non fiable : la fenêtre d'observation est
   coupée par les arrêts. C'est le risque technique central du requirement, à
   acter explicitement (voir §5 OPTION C / persistance).
5. **Faux positifs / faux négatifs** :
   - Le cache (`llm_semantic.py:216-219`) sert les analyses réussies sans appel
     OpenAI : pendant une panne OpenAI, une annonce déjà en cache ne produit pas
     de fallback → sous-comptage possible.
   - Un fallback isolé (un 429 ponctuel) n'est pas une panne. Compter en absolu
     génère du bruit. Il faut soit un seuil, soit un taux (fallbacks / analyses
     sur fenêtre), ce dernier étant plus juste mais nécessitant aussi le
     dénominateur (nombre d'analyses) — non compté aujourd'hui.
6. **Pas de rupture de contrat API** : aucune des options ne touche le schéma
   `/analyze` (anti-pattern CONTEXT §11.9 non concerné). Bon point.

## 5. OPTIONS chiffrées (choix structurants)

### 5.1 Approche de détection

**OPTION A — Externe (Better Stack / log drain Fly → webhook)**
- Principe : drainer les logs Fly vers un service qui parse `LLM call failed`
  et déclenche une alerte sur seuil.
- Avantages : agrège tous les logs, fenêtre temporelle fiable (indépendante de
  l'auto-stop), pas de code applicatif.
- Inconvénients : nouveau vendor + compte ; free tier Better Stack limité en
  rétention ; configuration log drain hors-repo (non versionnée, fragile) ;
  dépendance et surface d'attaque accrues ; le drain peut nécessiter une
  config DNS / un endpoint récepteur. Risque anti-pattern (§4.2, §4.3).
- Coût : free tier possible mais à surveiller ; effort de config ~1 h + compte.
- Adéquation MVP : faible. Sur-dimensionné pour le trafic actuel.

**OPTION B — In-app (compteur + endpoint admin, polling cron)**
- Principe : incrémenter un compteur au point de fallback
  (`llm_semantic.py:242`) ; exposer `GET /admin/health/llm` (protégé
  `X-Admin-Token`) renvoyant `{fallbacks, total_calls, window_start, last_failure_at}` ;
  un cron GitHub Actions (modèle `collect.yml`) interroge cet endpoint, compare
  au seuil, et notifie si dépassement.
- Avantages : zéro vendor, zéro coût marginal, tout dans le repo (versionné,
  testable via pytest), réutilise les patterns existants (admin token, cron).
- Inconvénients : compteur en mémoire **perdu à chaque auto-stop**
  (`fly.toml:13`) → fenêtre non fiable (§4.4) ; le polling réveille la VM
  (chaque GET la sort de l'auto-stop → micro-coût et fausse la mesure
  d'idle) ; nécessite de définir la fréquence de polling.
- Coût : ~0 €. Effort ~1-2 h.
- Adéquation MVP : bonne sur la légèreté, mais la fiabilité de la fenêtre est le
  point faible (cf. variante persistance).

**OPTION B' — In-app persisté (variante de B)**
- Principe : au lieu d'un compteur volatile, écrire une ligne en base SQLite à
  chaque fallback (table `LlmFailure` ou réutilisation/extension d'un modèle
  type `Feedback`, `models.py:54-70`), et calculer le taux/heure par requête SQL
  sur `created_at` côté endpoint.
- Avantages : survit à l'auto-stop (volume `/data` persistant, `fly.toml:24-26`)
  → fenêtre temporelle fiable, vrai taux/heure ; cohérent avec l'existant.
- Inconvénients : écrit en base sur un chemin d'erreur (acceptable, volume
  négligeable) ; +1 table + micro-migration ; léger surplus de code.
- Coût : ~0 € (volume déjà payé ~0,15 €/mois). Effort ~2 h.
- Adéquation MVP : meilleur compromis fiabilité/légèreté si l'on veut un vrai
  taux/heure. À mettre en balance avec la question « en a-t-on besoin ? » (§7).

**OPTION C — Hybride**
- Externe pour l'agrégation + in-app pour l'exposition. Cumule les
  inconvénients (vendor + code) sans bénéfice net à ce stade. Écartée a priori.

Recommandation analyste : **B' si l'on veut un signal fiable, B si l'on accepte
un signal grossier**, A et C écartées pour un MVP solo (anti-pattern vendor).
Décision à l'humain.

### 5.2 Canal de notification

- **Slack (webhook entrant)** : simple, pas de domaine requis, secret unique
  (URL webhook) stockable en repo secret GitHub. Recommandé.
- **Email** : bloqué par l'absence de domaine vérifié (même blocage que 9.2,
  CONTEXT §9.2). À écarter tant que le domaine n'est pas acquis.
- **GitHub issue auto-créée** : possible si le détecteur est un workflow Actions
  (token `GITHUB_TOKEN` natif, zéro secret externe, traçabilité dans le repo).
  Alternative crédible et 100 % gratuite, mais moins « temps réel » qu'un push
  Slack. Bon repli si l'on ne veut créer aucun webhook externe.

Recommandation analyste : **Slack** (push immédiat, prérequis minimal) ou
**GitHub issue** (zéro dépendance externe). Email écarté.

### 5.3 Persistance du signal
Voir OPTION B vs B' ci-dessus. Le cœur de l'arbitrage : un compteur volatile
suffit-il (détecte une rafale de fallbacks pendant que la VM est éveillée) ou
faut-il un vrai taux/heure résistant à l'auto-stop (table SQLite) ?

### 5.4 Définition du seuil et faux positifs
- En absolu : « N fallbacks dans la fenêtre » est simple mais sensible au
  trafic. À très faible trafic (§7), même 1-2 fallbacks peuvent passer
  inaperçus longtemps ou, à l'inverse, un seul appel raté = 100 % d'échec.
- En taux : « fallbacks / total_calls > X % sur fenêtre » est plus juste mais
  exige de compter aussi les succès (pas fait aujourd'hui ; à ajouter au point
  de succès `llm_semantic.py:275`).
- Distinguer panne OpenAI durable vs erreur ponctuelle : suggérer un seuil avec
  un minimum d'échantillon (ex. « ≥ K fallbacks ET ratio > X% sur 1 h »).
  Valeurs à fixer par l'humain (proposition : K=3, X=50%, fenêtre 60 min).

## 6. Challenge du requirement (posture adversariale)

1. **Est-ce le bon moment ?** Le MVP est en phase test, trafic quasi nul
   (CONTEXT §3.1, §4.3). Un fallback n'est observable que s'il y a des analyses.
   À zéro utilisateur, le monitoring ne se déclenchera jamais — il surveille un
   événement qui ne peut survenir que si quelqu'un analyse pendant une panne
   OpenAI. Le bénéfice réel est aujourd'hui faible. Priorité à challenger face à
   la dette plus structurante (tests, migration `on_event`→lifespan, cache LLM
   persistant — CLAUDE §11).
2. **Le requirement confond deux signaux** (« LLM call failed » OU `_FALLBACK ») :
   c'est un seul et même événement (§1.2). La rédaction surestime la complexité.
3. **L'option vendor (Better Stack) est sur-dimensionnée** pour un solo MVP : un
   compteur in-app + endpoint admin suffit techniquement et reste dans le repo.
   Le requirement liste le vendor en premier alors que c'est l'option la plus
   lourde.
4. **Plus simple encore que tout ce qui précède** : comme `analyze_semantic`
   logge déjà `LLM call failed`, un humain peut le voir via `fly logs` (CONTEXT
   §7.1). Le besoin réel à ce stade pourrait se réduire à : (i) un test de
   régression garantissant que le log de fallback existe et reste explicite
   (verrou anti-régression, esprit lessons.md), et (ii) un mémo « brancher une
   alerte quand le trafic décollera ». Investir 1-2 h dans une chaîne d'alerte
   complète maintenant peut être prématuré.
5. **Le polling réveille la VM** : un cron qui interroge un endpoint toutes les
   heures empêche partiellement l'auto-stop, ce qui va à l'encontre de la
   sécurité financière (CONTEXT §3.3 « Fly auto-stop = pas de facture qui
   dérape »). Effet probablement négligeable mais à noter.

Position analyste : le requirement est légitime à terme mais probablement
**prématuré au trafic actuel** ; si on le fait, retenir l'option la plus légère
(in-app, sans vendor), idéalement réduite au strict nécessaire.

## 7. QUESTIONS POUR L'HUMAIN (GATE 1)

1. **[ARBITRAGE] Faut-il faire 9.3 maintenant ?** Au trafic actuel (quasi nul),
   un fallback n'est presque pas observable. Options : (a) reporter et se
   contenter d'un test de régression sur le log `LLM call failed` + mémo ; (b)
   livrer une chaîne d'alerte légère dès maintenant. Reco analyste : (a), ou (b)
   en version minimale uniquement.

2. **[ARBITRAGE] Approche de détection.** A externe (Better Stack / log drain),
   B in-app compteur volatile + endpoint admin, B' in-app persisté (table
   SQLite), C hybride. Reco analyste : **B'** si l'on veut un taux/heure fiable
   malgré l'auto-stop, **B** si un signal grossier suffit ; **A et C écartées**
   (vendor sur-dimensionné pour MVP < 1 €/mois). Point dur à acter : le compteur
   en mémoire est **perdu à chaque auto-stop Fly** (`fly.toml:13`), donc B seul
   ne fournit pas de fenêtre fiable.

3. **[ARBITRAGE] Persistance.** Compteur volatile (B) vs table SQLite (B',
   modèle `Feedback` à imiter, `db/models.py:54-70`). Reco analyste : table
   SQLite si l'alerte doit être fiable ; sinon ne pas sur-investir.

4. **[ARBITRAGE] Canal de notification.** Slack (webhook, pas de domaine),
   GitHub issue auto (zéro dépendance externe), email (bloqué : pas de domaine,
   cf. 9.2). Reco analyste : **Slack** ou **GitHub issue** ; email écarté.

5. **[ARBITRAGE] Définition du seuil.** Absolu (N fallbacks/fenêtre) vs taux
   (fallbacks/total_calls). Le taux exige de compter aussi les succès (ajout au
   point de succès `llm_semantic.py:275`). Reco analyste : taux avec garde
   d'échantillon, ex. « ≥ 3 fallbacks ET > 50 % sur 60 min » — valeurs à
   confirmer.

6. **[ARBITRAGE] Endpoint de santé : périmètre exposé.** Confirmer que
   `GET /admin/health/llm` (protégé `X-Admin-Token`) ne renvoie que des agrégats
   (compteur, fenêtre, `last_failure_at`), jamais le contenu d'annonce ni le
   message d'exception OpenAI. Reco analyste : agrégats stricts uniquement.

7. **[ACTION EXTERNE] Créer un webhook Slack entrant** (si canal Slack retenu),
   et le stocker en secret repo GitHub (ex. `SLACK_WEBHOOK_URL`) si la détection
   est un cron Actions, ou en `fly secrets` si la notification part du backend.
   Action à réaliser par l'humain (création de l'app/webhook Slack).

8. **[ACTION EXTERNE] (uniquement si OPTION A retenue)** Créer un compte Better
   Stack / Logtail et configurer un log drain Fly. Reco analyste : ne pas faire
   (option écartée) sauf décision contraire en Q2.
