# 9.3 — SPEC : monitoring qualité IA (version minimale, alerte différée)

> Issue du GATE 1 sur `docs/specs/9.3-ANALYSE.md`. Décision humaine (2026-06-04) :
> au trafic quasi nul, livrer le **strict nécessaire** — un verrou de régression
> sur le marqueur de fallback — et **différer la chaîne d'alerte** jusqu'à ce que
> le trafic la justifie.

## 1. Livré dans ce ticket

- **Test de régression** `backend/tests/test_llm_fallback.py` :
  force l'appel OpenAI à échouer (monkeypatch) et vérifie que
  `analyze_semantic` (1) émet le log **`LLM call failed`** au niveau **ERROR**
  sur le logger **`llm_semantic`**, et (2) renvoie bien `_FALLBACK`
  (`summary == "Analyse indisponible."`, listes vides).
- **Mémo** dans `CONTEXT.md` §8 (dette/observabilité) : brancher la vraie alerte
  quand le trafic décollera.

Pourquoi ce périmètre : le fallback LLM est émis en **un seul point**
(`backend/app/llm_semantic.py:236-242`). Le « log warning » et le « retour
`_FALLBACK` » du requirement sont le **même événement**. Le seul signal
loggable est la chaîne `LLM call failed` (la variable `_FALLBACK` n'apparaît
jamais dans les logs). Verrouiller ce marqueur garantit qu'un futur monitoring
aura toujours un signal stable sur lequel s'appuyer.

## 2. Différé (chaîne d'alerte complète) — backlog

À reprendre via l'atelier quand le trafic le justifiera. Conception cible déjà
arbitrée par l'analyse (option B') :

- **Détection persistée** : table SQLite `LlmFailure` (modèle `Feedback` à
  imiter) écrite au point de fallback → taux/heure **fiable malgré l'auto-stop
  Fly** (un compteur en mémoire est remis à zéro à chaque réveil de VM,
  `fly.toml` `auto_stop_machines=true`).
- **Exposition** : `GET /admin/health/llm` protégé par `X-Admin-Token`,
  renvoyant des agrégats uniquement (jamais de contenu d'annonce ni de message
  d'exception brut).
- **Alerte** : cron GitHub Actions (modèle `collect.yml`) qui interroge
  l'endpoint et, au-delà d'un seuil (proposition : ≥ 3 fallbacks ET > 50 % sur
  60 min), crée une **issue GitHub** (zéro secret externe, `GITHUB_TOKEN`) ou
  pousse sur un **webhook Slack**. Email **écarté** (bloqué par l'absence de
  domaine, cf. 9.2).
- **Vigilance** : le polling réveille la VM (va à l'encontre de l'auto-stop) ;
  le cache LLM peut masquer une panne (annonces déjà en cache servies sans appel).

## 3. Hors-périmètre

- Pas de vendor (Better Stack / log drain Fly) : sur-dimensionné pour un MVP solo.
- Pas de monitoring d'erreurs généraliste (Sentry) : autre ligne de dette.
- Pas d'alerte de coût OpenAI : c'est 9.4 (config UI, hors code).

## 4. Critères d'acceptation (version minimale)

1. `test_llm_fallback.py` passe : le marqueur `LLM call failed` (ERROR, logger
   `llm_semantic`) est émis quand l'appel OpenAI échoue, et `_FALLBACK` est
   renvoyé.
2. Le test échouerait si le message de log changeait ou disparaissait
   (verrou de régression effectif).
3. Aucun changement de comportement de prod (test-only + mémo doc).
