## 1. Objectif et périmètre

### 1.1 Reformulation de l'objectif
Publier sur le frontend Vercel du contenu pédagogique répondant aux questions
que se posent les acheteurs ("Comment savoir si une annonce immobilière est
honnête ?"), dans le but de capter du trafic SEO de longue traîne, en cohérence
avec le positionnement : factuel, prudent, non-commercial, pas d'estimation de
prix, pas de conseil juridique (CONTEXT §1.2, §1.4, §2.1, §11). Le requirement
propose un agent Claude qui produit une liste de ~20 questions d'acheteurs puis
rédige ~1 article/semaine, publié en pages MDX sous `frontend/app/blog/`.

### 1.2 Ce que le requirement suppose vs l'état réel (ancrage code)
Le requirement suppose trois choses qui ne sont PAS vraies aujourd'hui et
qu'il faut acter :

1. **Aucune route blog n'existe.** L'App Router ne contient que
   `frontend/app/globals.css`, `frontend/app/layout.tsx`,
   `frontend/app/page.tsx`. Pas de `frontend/app/blog/`. Tout est à créer.
2. **MDX n'est pas configuré.** `frontend/package.json:10-13` ne liste que
   `next`, `react`, `react-dom` — pas de `@next/mdx`, `@mdx-js/loader`,
   `@mdx-js/react`. `frontend/next.config.js` est minimal (`reactStrictMode:
   false` uniquement, pas de `withMDX`, pas de `pageExtensions`). Activer MDX
   = nouvelle(s) dépendance(s) + modification de `next.config.js`, à rebours du
   minimalisme assumé du projet (CONTEXT §12 « Pas de Tailwind », CLAUDE §10
   « CSS vanilla, pas de lib UI »).
3. **Le socle SEO de base est absent.** Recherche `robots|sitemap|
   generateMetadata|opengraph|metadataBase` sur `frontend/` : **aucune
   occurrence**. `frontend/app/layout.tsx:3-6` ne porte qu'un `metadata`
   statique (title + description), sans `metadataBase`, sans Open Graph, sans
   template de titre. Aucun `public/` (pas de `robots.txt`, pas de favicon).
   CONTEXT §8 (dette) liste explicitement « Robots.txt côté frontend » et
   « Sitemap pour SEO » comme NON faits. Conséquence directe : des articles
   MDX publiés sans sitemap, sans robots, sans métadonnées par page ne seront
   pas correctement indexés — l'objectif « ramener du trafic » ne serait pas
   atteint par le seul scaffold MDX.

### 1.3 In / Out (proposition à arbitrer en GATE 1)

In (cœur du requirement, version défendable) :
- Un scaffold de blog côté Next.js (route, rendu d'article, style cohérent avec
  le design system « Cohérence »).
- Le socle SEO minimal sans lequel le blog est inutile : `metadataBase` +
  `generateMetadata` par article, `app/sitemap.ts`, `app/robots.ts` (API
  natives Next.js, zéro dépendance).
- Au moins 1 article pilote, rédigé et **revu par un humain**.
- Optionnellement : un agent qui propose des sujets/plans (liste de ~20
  questions d'acheteurs).

Out (à exclure sauf arbitrage contraire) :
- L'automatisation de la **rédaction** (agent qui écrit et publie sans revue) —
  risque éditorial/légal majeur, voir §4.
- La cadence « 1 article/semaine automatique » avant d'avoir validé l'hypothèse
  d'acquisition SEO (CONTEXT §2.4 : hypothèse NON validée).
- Un CMS headless (Contentful, Sanity…) — sur-dimensionné (voir §5).
- Données structurées avancées, maillage interne élaboré, AMP, etc. (peuvent
  venir après si l'hypothèse se valide).

## 3. Dépendances et ordre

- **Le SEO socle (robots/sitemap/metadata) est un prérequis fonctionnel** du
  blog, pas une option : sans lui, l'objectif « trafic » n'est pas atteignable.
  Ces trois éléments sont par ailleurs déjà de la dette identifiée (CONTEXT §8),
  donc 9.5 est l'occasion logique de la solder. Ordre proposé : socle SEO +
  scaffold AVANT toute automatisation.
- **L'automatisation de la rédaction (agent + cron) dépend** d'un scaffold et
  d'un processus de revue déjà en place. La faire en premier serait inverser
  l'ordre de risque.
- **Prérequis manquant : l'URL frontend Vercel de production.** Nécessaire pour
  `metadataBase` (URLs canoniques, Open Graph, sitemap). CONTEXT §10 indique
  qu'elle n'est pas connue dans la doc. À fournir (action externe, §7).
- **Lien avec 9.2 (email) : aucun.** Le blog est du contenu statique, pas
  d'envoi. En revanche, un domaine custom (non acquis, CONTEXT §3.2) renforcerait
  le SEO et l'autorité de domaine ; ce n'est pas bloquant pour publier sur
  `*.vercel.app`, mais à noter pour l'efficacité réelle.
- **Aucune nouvelle clé API requise** pour le rendu. Une éventuelle génération
  par agent Claude réutiliserait l'outillage Claude Code / une clé Anthropic en
  CI (secret repo), pas la clé OpenAI du backend.

## 4. Risques d'anti-pattern

1. **Risque éditorial/légal MAJEUR de la rédaction automatisée** (CONTEXT §1.2,
   §1.4, §2.1, §11 ; CLAUDE §1). Le positionnement interdit : estimation de prix,
   conseil juridique, données inventées, ton commercial. Un article généré et
   publié sans revue humaine peut, sur un sujet « comment négocier » ou
   « l'annonce est-elle honnête », glisser vers du conseil juridique implicite,
   une promesse d'estimation, ou une affirmation non sourcée — exactement les
   anti-patterns interdits. **Tout contenu généré doit passer une revue humaine
   avant publication.** La publication directe par un workflow est à proscrire.
2. **Contenu trompeur / EAT SEO.** Du contenu LLM de masse, faible qualité et non
   relu, est pénalisé par Google (Helpful Content) et abîme la marque « factuel,
   prudent, rassurant » (CONTEXT §2.1). Le risque réputationnel dépasse le gain
   SEO hypothétique.
3. **Nouvelle dépendance à rebours du minimalisme.** Ajouter `@next/mdx` + 2-3
   paquets va contre la ligne « dépendances UI minimales » (CONTEXT §12, CLAUDE
   §10). À ne faire que si MDX apporte un bénéfice réel vs alternatives sans
   dépendance (§5).
4. **Sur-investissement sur une hypothèse non validée.** L'acquisition SEO est
   listée comme hypothèse NON validée (CONTEXT §2.4) ; le SEO de longue traîne
   met des mois à porter. Bâtir une usine à contenu automatisée avant le premier
   signal de trafic est un classique de sur-ingénierie pour un MVP < 1 €/mois.
5. **Coût.** Le rendu statique sur Vercel free tier reste à 0 € (CONTEXT §3.2).
   Une génération hebdo par agent a un coût LLM négligeable, mais le **vrai coût
   est humain** (revue éditoriale de chaque brouillon) — à ne pas sous-estimer.
6. **Pas de redistribution d'annonces.** Les articles ne doivent contenir aucune
   annonce brute ni capture de portail (anti-pattern CONTEXT §11.3) ; rester sur
   du pédagogique générique. Risque faible si la consigne est dans le prompt et
   vérifiée en revue.
7. **Faux ami « MDX = SEO ».** MDX ne donne aucun SEO en soi. Le SEO vient des
   métadonnées par page, du sitemap, du robots, du contenu de qualité et du
   maillage — pas du format de fichier. Ne pas confondre l'effort technique
   (scaffold) avec l'effort produit (qualité + socle SEO).

## 5. OPTIONS chiffrées (choix structurants)

### 5.1 Mécanique de rendu des articles

**OPTION R1 — Composants `.tsx` purs (zéro dépendance)**
- Principe : chaque article est un server component React sous
  `frontend/app/blog/<slug>/page.tsx`, le corps en JSX stylé avec les tokens
  `globals.css` ; métadonnées via `export const metadata`/`generateMetadata`.
- Avantages : zéro dépendance ajoutée (respecte CONTEXT §12), 100 % statique,
  SEO maximal, contrôle total du style. Aligné avec l'existant (`page.tsx` est
  déjà du React).
- Inconvénients : écrire de la prose en JSX est moins agréable qu'en Markdown ;
  un agent générant du `.tsx` doit produire du JSX valide (plus fragile qu'un
  `.md`).
- Coût : 0 € / 0 dépendance. Effort scaffold ~2-3 h.

**OPTION R2 — Markdown rendu sans MDX (`.md` + parseur léger)**
- Principe : articles en `.md` (front-matter + corps), lus au build par un petit
  parseur Markdown (ex. un convertisseur Markdown→HTML minimal) et rendus en
  statique ; pas d'exécution de composants dans le contenu.
- Avantages : rédaction confortable en Markdown (idéal pour un agent : produit du
  texte, pas du code), pas de couplage `next.config`, dépendance unique et légère
  (un parseur MD) plutôt que la chaîne MDX complète.
- Inconvénients : 1 dépendance quand même (parseur) ; pas de composants
  interactifs dans l'article (non nécessaire pour du pédagogique).
- Coût : 0 € hébergement ; 1 petite dépendance. Effort scaffold ~3-4 h.

**OPTION R3 — MDX via `@next/mdx` (ce que propose le requirement)**
- Principe : configurer `withMDX` + `pageExtensions` ; articles en `.mdx`
  mêlant Markdown et composants React.
- Avantages : standard Next.js documenté ; permet d'insérer des composants
  (ex. un encart « disclaimer », un CTA vers l'analyseur) dans la prose.
- Inconvénients : 3 dépendances (`@next/mdx`, `@mdx-js/loader`, `@mdx-js/react`)
  + modification `next.config.js` + `mdx-components.tsx` ; surface de maintenance
  et de compatibilité (Next 16) plus large ; à rebours du minimalisme. Bénéfice
  des composants-dans-la-prose faible pour du contenu pédagogique simple.
- Coût : 0 € hébergement ; 3 dépendances. Effort scaffold ~4-5 h + risque de
  friction de config.

**OPTION R4 — CMS headless** (Contentful/Sanity/etc.)
- Sur-dimensionné pour un MVP solo : nouveau vendor, compte, surface, coût
  potentiel, et inutile à ce volume. **Écartée a priori.**

Recommandation analyste : **R1** (composants `.tsx` purs) pour rester à zéro
dépendance et au SEO maximal, ou **R2** si l'on veut une rédaction en Markdown
plus confortable pour l'agent. **R3 (MDX)** seulement si l'on tient à insérer des
composants riches dans la prose — bénéfice faible vs le coût en dépendances.
**R4 écartée.** Décision à l'humain.

### 5.2 Rôle de l'agent Claude

**OPTION A1 — Agent « idéation/plan » + rédaction humaine**
- L'agent produit la liste de ~20 questions d'acheteurs et des plans d'article
  (titres, angles, points à couvrir). L'humain (ou un second passage assisté)
  rédige et valide.
- Avantages : risque éditorial/légal maîtrisé ; qualité et ton garantis ; aligné
  positionnement. Coût LLM négligeable.
- Inconvénients : moins « automatique » que le requirement ne l'imagine ; débit
  d'articles limité par le temps humain.

**OPTION A2 — Agent « rédaction » + revue humaine obligatoire avant publication**
- L'agent rédige un brouillon complet ; un workflow planifié **ouvre une PR de
  brouillon** (patron `collect.yml`, mais ouverture de PR, pas merge) ; un humain
  relit, corrige, merge → Vercel publie au merge sur `main`.
- Avantages : débit plus élevé ; la revue humaine reste le garde-fou (pas de
  publication directe) ; traçabilité Git (diff = brouillon revu).
- Inconvénients : charge de revue récurrente (le vrai coût) ; risque que la revue
  devienne tampon si la cadence pousse ; nécessite une clé Anthropic en CI.

**OPTION A3 — Rédaction automatique + publication directe (ce que le mot
« 1 article/semaine → publication » laisse entendre)**
- **Écartée** : viole le garde-fou de revue humaine (§4.1). Risque légal/éditorial
  inacceptable vu le positionnement.

Recommandation analyste : **A1 pour démarrer** (idéation + rédaction humaine du
pilote), puis **A2 (PR de brouillon, jamais de merge auto)** seulement si
l'hypothèse SEO montre un signal et qu'on veut monter en cadence. **A3 proscrite.**

### 5.3 Périmètre du socle SEO (non optionnel)
- Minimal indispensable : `metadataBase` + `generateMetadata` par article
  (title/description/canonical/OG), `app/sitemap.ts`, `app/robots.ts` — tous via
  API natives Next.js, **zéro dépendance**, ~1-2 h. Solde aussi la dette CONTEXT
  §8. Sans cela, l'objectif trafic est inatteignable.
- Optionnel (différable) : données structurées (JSON-LD Article), maillage
  interne, breadcrumbs.

### 5.4 Périmètre MVP / cadence
- **OPTION P1 — Pilote** : socle SEO + scaffold (R1 ou R2) + **1 article pilote
  rédigé et revu humainement** + liste de 20 sujets par l'agent (A1).
  Automatisation **différée**. Permet de tester l'hypothèse à coût minimal.
- **OPTION P2 — Usine** : tout P1 + workflow planifié de brouillons (A2) dès
  maintenant.
- Recommandation analyste : **P1**. On ne construit l'usine (P2) qu'après un
  premier signal de trafic, conformément à « ne pas sur-investir sur une
  hypothèse non validée » (CONTEXT §2.4).

## 6. Challenge du requirement (posture adversariale)

1. **Est-ce prioritaire maintenant ?** Le MVP est en phase de test produit, à
   trafic quasi nul (CONTEXT §3.1, §4.3) et avec de la dette plus structurante
   (tests pytest, migration `on_event`→lifespan, cache LLM persistant — CLAUDE
   §11 ; peupler/valider le pilier prix — CONTEXT §8). L'acquisition SEO est une
   hypothèse NON validée (§2.4) et le SEO de longue traîne met **des mois** à
   porter. Investir dans une usine à contenu avant d'avoir un seul signal est
   probablement prématuré. Le bouche-à-oreille (§2.4) porte plus vite à ce stade.
2. **Le requirement confond format et SEO.** « publication sur Vercel (pages
   MDX) » suggère que MDX = trafic. Faux : le trafic vient du socle SEO (absent
   aujourd'hui) et de la qualité, pas du format de fichier. Le vrai travail
   (métadonnées par page, sitemap, robots, qualité, maillage) n'est pas nommé
   dans le requirement alors qu'il est la condition de l'objectif.
3. **L'automatisation de la rédaction est le point le plus risqué et le moins
   nécessaire.** « 1 article/semaine » automatique entre en tension directe avec
   le positionnement juridiquement prudent. Un agent qui **propose des sujets**
   est sûr et utile ; un agent qui **rédige et publie** est dangereux. Si
   automatisation il y a, elle doit ouvrir une PR de brouillon, jamais merger.
4. **MDX va contre le minimalisme du projet** (CONTEXT §12, CLAUDE §10). Des
   composants `.tsx` purs (R1) ou du Markdown léger (R2) atteignent le même
   résultat SEO sans la chaîne MDX. Le requirement choisit l'option la plus
   lourde par défaut.
5. **Version minimale qui teste l'hypothèse** : socle SEO (solde la dette §8) +
   scaffold sans dépendance lourde + **1 article pilote relu** + une liste de 20
   sujets proposée par l'agent. Mesurer l'indexation et le trafic pendant
   quelques semaines AVANT d'investir dans l'automatisation. C'est la
   recommandation P1 + R1/R2 + A1.

Position analyste : requirement légitime sur le fond (le contenu pédagogique
sert la marque), mais **mal posé** (MDX et rédaction auto mis au centre, socle
SEO et revue humaine oubliés) et probablement **prématuré à pleine échelle**. À
réduire à un pilote sobre, automatisation différée.

## 7. QUESTIONS POUR L'HUMAIN (GATE 1)

1. **[ARBITRAGE] Faut-il faire 9.5 maintenant, et à quelle échelle ?** Options :
   (P1) pilote — socle SEO + scaffold + 1 article relu + 20 sujets proposés,
   automatisation différée ; (P2) usine — P1 + workflow de brouillons dès
   maintenant ; (différer) prioriser dette/pilier prix d'abord. Reco analyste :
   **P1** (hypothèse SEO non validée, CONTEXT §2.4 ; SEO long-tail lent).

2. **[ARBITRAGE] Mécanique de rendu.** R1 composants `.tsx` purs (0 dépendance),
   R2 Markdown léger (1 petite dépendance), R3 MDX `@next/mdx` (3 dépendances +
   `next.config`), R4 CMS (écarté). Reco analyste : **R1** (ou R2 si rédaction
   Markdown souhaitée) ; **R3 seulement** si insertion de composants dans la
   prose jugée nécessaire ; R4 écartée. Note : R3 va à rebours du minimalisme
   (CONTEXT §12, CLAUDE §10).

3. **[ARBITRAGE] Rôle de l'agent Claude.** A1 idéation/plan + rédaction humaine ;
   A2 rédaction + PR de brouillon avec revue humaine obligatoire (jamais de merge
   auto) ; A3 rédaction + publication directe (proscrite). Reco analyste : **A1
   pour démarrer**, **A2 (PR jamais merge auto)** uniquement si montée en cadence
   après signal de trafic. **A3 proscrite** (risque éditorial/légal, §4.1).

4. **[ARBITRAGE] Périmètre du socle SEO.** Confirmer qu'il est inclus dans 9.5
   (sinon l'objectif trafic est inatteignable) : `metadataBase` +
   `generateMetadata` par article + `app/sitemap.ts` + `app/robots.ts` (API
   natives Next.js, 0 dépendance ; solde la dette CONTEXT §8). Données
   structurées / maillage : différables. Reco analyste : **inclure le socle
   minimal**, différer le reste.

5. **[ARBITRAGE] Garde-fou éditorial.** Acter qu'aucun article généré n'est
   publié sans revue humaine, et que le prompt agent interdit explicitement :
   estimation de prix, conseil juridique, données inventées, ton commercial,
   redistribution d'annonces (CONTEXT §1.2, §1.4, §11). Reco analyste : règle
   ferme + consigne dans le prompt + vérification en revue.

6. **[ACTION EXTERNE] Fournir l'URL frontend Vercel de production**
   (manquante, CONTEXT §10) — nécessaire pour `metadataBase`, canonicals,
   sitemap, Open Graph. Sans elle, le socle SEO ne peut pas être correctement
   configuré.

7. **[ACTION EXTERNE — conditionnelle, si A2 retenue]** Fournir une clé API
   Anthropic en secret repo GitHub pour le workflow de génération de brouillons
   (patron `collect.yml`, mais ouverture de PR — pas de merge automatique). Non
   requise pour P1 + A1.

8. **[ARBITRAGE — secondaire] Domaine custom.** Un domaine propre (non acquis,
   CONTEXT §3.2) renforcerait l'autorité SEO vs `*.vercel.app`. Non bloquant pour
   publier, mais impacte l'efficacité réelle de l'objectif « trafic ». Reco
   analyste : noter comme amélioration, hors périmètre 9.5.
