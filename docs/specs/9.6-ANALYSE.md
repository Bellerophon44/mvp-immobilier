## 1. Objectif et perimetre

### 1.1 Reformulation
Permettre a un utilisateur d'**enregistrer une ou plusieurs annonces (par lien)**
dans une liste de suivi, puis, via un **cron quotidien**, re-fetcher chaque URL,
**comparer le contenu** a l'etat precedent (hash), et si le contenu a change —
notamment une **baisse de prix** — **notifier l'utilisateur par email** ("Cette
annonce a baisse de X EUR"). But produit : engagement/retention, capter l'intention
d'achat dans la duree.

Stack suggeree par le requirement : table `Watchlist(id, user_email, url,
last_hash, last_price, created_at)`, reuse de `url_fetch.py`, cron type
`collect.yml`. Effort annonce : 4-6h. Prerequis annonce : **auth utilisateur (au
moins email magic link)**.

### 1.2 Mise en garde de cadrage immediate
Le requirement **melange deux operations de cout tres different** et il faut les
separer des le cadrage :
- **Detecter un changement** = re-fetch + hash du contenu (via `url_fetch.py`).
  Cout quasi nul (une requete HTTP, un `sha256`). Pas de LLM.
- **Re-analyser** = re-appel LLM (`run_full_analysis`) sur le nouveau contenu.
  Cout ~0,001 EUR/annonce/jour + latence + risque fallback.

Le titre dit "re-analyse" mais la mecanique decrite (hash + baisse de prix) ne
necessite **pas** de LLM pour fonctionner. C'est un point d'arbitrage central
(Q6), pas un detail d'implementation.

### 1.3 Dans le perimetre (IN), si le requirement etait fait tel quel
- Une table de persistance des URLs suivies par utilisateur.
- Un mecanisme d'authentification permettant de rattacher une watchlist a un
  utilisateur (le requirement impose "au moins email magic link").
- Un cron quotidien (GitHub Actions, modele `collect.yml`) qui itere les URLs.
- Un detecteur de changement (hash) et, si retenu, une extraction de prix.
- Un envoi d'email de notification (et, pour le magic link, un envoi d'email
  d'authentification).
- Conformite RGPD d'un traitement **recurrent** de donnees personnelles.

### 1.4 Hors perimetre (OUT) — a confirmer
- Pas d'historique de prix consultable en UI (graphe d'evolution) : le requirement
  ne le demande pas (juste `last_price`). A confirmer (un seul `last_price` ne
  permet pas un historique).
- Pas de gestion fine des preferences de notification (frequence, mute par annonce).
- Pas de dashboard "mes annonces suivies" cote front au-dela d'un minimum.
- Pas de support des portails anti-bot (Leboncoin/SeLoger/Bien'ici en HTML) :
  techniquement hors de portee du fetch actuel (cf. §4).

## 2. Cartographie de l'impact reel dans le code

### 2.1 Ce qui existe et serait reutilise
- `backend/app/url_fetch.py:49-88` — `fetch_listing_text(url)` : fetch + extraction
  texte. Reutilisable pour le hash de contenu. **Limites importantes** :
  - filtre SSRF minimal (`_is_safe_url`, lignes 28-46) — OK pour des URLs publiques.
  - plafond `MAX_TEXT_LENGTH = 8000` (ligne 24) : le hash porterait sur le texte
    **tronque**, pas la page entiere — acceptable mais a noter.
  - **ne renvoie PAS le prix** : il extrait du texte brut. Aucune extraction
    structuree de prix ici (le prix est extrait plus loin, par le LLM, cf. §2.4).
  - retourne `None` sur echec/anti-bot/contenu trop court — frequent sur grands
    portails (cf. CONTEXT §4.3).
- `backend/db/models.py:54-71` (`Feedback`) + `db/session.py:35-61` — **patron** de
  nouvelle table + micro-migration idempotente (`_ADD_COLUMNS`, `ALTER TABLE ADD
  COLUMN`, `create_all`). Une nouvelle table `Watchlist` se creerait via
  `create_all` au demarrage (`init_db`, `main.py:50-52`).
- `backend/app/main.py:119-124` (`_check_admin_token`) — **patron** d'auth par
  secret en temps constant (`hmac.compare_digest`) + garde-fou 503. C'est le seul
  mecanisme d'"auth" du repo aujourd'hui, et il est **machine-a-machine** (CI ->
  admin), pas utilisateur. Il n'est PAS reutilisable tel quel pour de l'auth
  utilisateur (voir §3.1).
- `backend/jobs/push_comparables.py:60-101` — **patron** de job CI qui POST vers un
  endpoint admin protege par `X-Admin-Token`. Un cron de watchlist suivrait
  exactement ce modele (collecte cote runner, ecriture cote backend).
- `.github/workflows/collect.yml:1-33` — **patron** de cron quotidien/manuel
  (`schedule: cron`, `workflow_dispatch`, secret `ADMIN_TOKEN`). Directement
  duplicable pour un `watchlist-check.yml`.
- `backend/app/analysis.py` (`run_full_analysis`) + `llm_semantic.py` — la
  re-analyse LLM, si retenue, reutilise l'orchestrateur existant (le prix
  `price_total` est extrait par le LLM dans `listing`, cf. CLAUDE.md §6).

### 2.2 Ce qui n'existe PAS et serait a creer
- **Aucune brique d'authentification utilisateur.** Recherche dans le repo : aucune
  table user, aucun token de session, aucun magic link, aucun cookie d'auth, aucune
  notion de "compte". Le seul header d'auth est `X-Admin-Token` (M2M). C'est un
  chantier neuf et entier (§3.1).
- **Aucun envoi d'email.** Confirme par la spec 9.2 (`docs/specs/9.2-ANALYSE.md`) :
  pas de provider, pas de domaine verifie, pas de `RESEND_API_KEY`. 9.2 a ete
  **differe pour cette raison precise**.
- **Aucune extraction de prix hors LLM.** Detecter une "baisse de prix" de facon
  fiable sur une page arbitraire necessiterait soit le LLM (cher, recurrent), soit
  un parseur de prix par site (n'existe que dans les `scrapers/sources/*`, pour des
  sites precis, pas pour une URL quelconque saisie par l'utilisateur).

### 2.3 Schema `/analyze`
- Aucun impact direct : 9.6 n'a pas besoin de modifier `/analyze`. Les nouveaux
  endpoints (enregistrer/lister/supprimer une annonce suivie, magic link) sont
  independants. La regle "ne pas casser `/analyze` sans MAJ `lib/api.ts`"
  (lessons.md, CONTEXT §11.9) n'est donc pas en cause **sauf** si on choisit de
  greffer l'enregistrement dans `/analyze` (deconseille).

### 2.4 DB
- Nouvelle table `Watchlist` (et probablement une table de tokens magic link, et
  une table users selon le design d'auth). Micro-migration via `create_all` au
  demarrage (patron `db/session.py`).
- **SQLite mono-writer sur volume Fly + auto-stop** (`fly.toml:9-26`,
  CONTEXT §12) : le cron CI ecrirait via un endpoint backend (comme
  `push_comparables`), pas en direct sur le fichier. Coherent avec l'archi
  existante (1 machine, 1 volume, `min_machines_running = 0`).

### 2.5 Frontend
- `frontend/lib/api.ts` — nouvelles fonctions (`saveWatch`, `listWatch`, auth magic
  link) sur le modele de `sendFeedback` (lignes 90-104, non bloquant) — mais
  l'auth introduit un etat de session a gerer cote client (token/cookie), ce que le
  front ne fait pas du tout aujourd'hui (`api.ts` est sans etat).
- `frontend/app/page.tsx` — UI "suivre cette annonce" + ecran de connexion magic
  link + eventuel "mes annonces". C'est plus qu'un bouton : c'est un parcours
  authentifie, etranger au front actuel (mono-page sans compte).

### 2.6 CI / Fly
- Nouveau workflow `watchlist-check.yml` (cron quotidien, modele `collect.yml`).
- `fly.toml` : `auto_stop_machines = true` + `min_machines_running = 0` — un cron
  quotidien **reveille la machine** (auto_start), comme la collecte hebdo. Cout
  marginal negligeable (CONTEXT §3.2). Point **non bloquant**.

## 3. Dependances et ordre — les DEUX prerequis bloquants

9.6 ne peut pas etre developpe-livre tant que ces deux chantiers n'existent pas.
Ils ne sont pas "inclus dans les 4-6h" : ce sont des chantiers a part entiere.

### 3.1 PREREQUIS BLOQUANT 1 — AUTH UTILISATEUR (chantier entier, pas un detail)
Le requirement ecrit "pre-requis : auth utilisateur (au moins email magic link)"
comme une ligne. En realite, **aucune auth utilisateur n'existe dans le repo**
(verifie : seul `X-Admin-Token` M2M existe). Un "magic link" implique concretement :

1. **Un envoi d'email** pour delivrer le lien (donc le MEME blocage que 9.2 :
   domaine verifie + provider + secret ; cf. §3.2). Le magic link **ne contourne
   pas** le besoin d'email, il l'**exige**.
2. **Generation et stockage de tokens** a usage unique, expirants, non devinables
   (table tokens, hash en base, TTL, invalidation apres usage).
3. **Une route de demande de lien** (`POST /auth/request`) + **une route de
   verification** (`GET /auth/verify?token=...`) qui ouvre une session.
4. **Une notion de session** : emettre et valider un cookie/JWT, le proteger
   (`HttpOnly`, `Secure`, `SameSite`), gerer l'expiration, le CORS avec
   credentials (le backend a deja `allow_credentials=True`, `main.py:44`, mais
   aucune session a porter).
5. **Securite** : rate limit sur la demande de lien (sinon relais de spam, comme
   9.2 §4.2 — or **aucun rate limit n'existe**, CONTEXT §8), anti-enumeration
   d'emails, protection CSRF selon le mecanisme de session.
6. **Cote front** : un parcours de connexion (saisie email -> "verifiez vos mails"
   -> clic -> session), un etat d'authentification persistant, la gestion de la
   deconnexion. Le front actuel est mono-page sans aucune notion de compte.

C'est un chantier de plusieurs heures a lui seul, transversal back+front, avec ses
propres risques de securite. **Il est sous-estime par la formulation du
requirement.** Alternative plus legere a arbitrer : auth deleguee (OAuth Google) ou
un service tiers (Clerk/Supabase Auth) — mais cela introduit un **nouveau vendor**
(anti-pattern minimalisme) et/ou de nouvelles donnees perso. A remonter (Q3).

### 3.2 PREREQUIS BLOQUANT 2 — ENVOI D'EMAIL (herite de 9.2, deux fois)
9.6 a besoin d'envoyer des emails pour **deux** usages distincts :
- le **magic link** (authentification) ;
- la **notification de baisse de prix** (la fonctionnalite elle-meme).

Or l'envoi d'email est **deja bloque et differe** par 9.2 pour une raison non
codable (`docs/specs/9.2-ANALYSE.md` §3.1) :
- **pas de domaine expediteur acquis** (CONTEXT §3.2 : "Domaine custom : non
  acquis") — sans domaine verifie (SPF/DKIM), les mails partent en spam ou sont
  refuses ; le domaine de test Resend n'envoie qu'a soi-meme.
- pas de compte provider, pas de secret `RESEND_API_KEY` pose sur Fly.

Donc **9.6 herite du blocage de 9.2 deux fois.** Tant que 9.2 (brique email) n'est
pas debloque cote infra, 9.6 ne peut etre ni teste de bout en bout (magic link), ni
livre (notification).

### 3.3 Ordre impose
```
[domaine + provider email (action humaine, prerequis de 9.2)]
        |
        v
[brique email reutilisable (9.2)]
        |
        v
[brique auth utilisateur (chantier neuf, magic link => depend de l'email)]
        |
        v
[9.6 watchlist + cron + notification]
```
9.6 est donc en **bout de chaine** : il depend d'au moins deux chantiers amont dont
l'un (email) est deja explicitement reporte, et l'autre (auth) n'existe pas du tout.

### 3.4 Prerequis manquants signales
- Domaine email : **manquant** (bloquant prod, partage avec 9.2).
- Auth utilisateur : **inexistante** (chantier neuf).
- Rate limit : **inexistant** (CONTEXT §8) — necessaire pour la demande de magic
  link ET pour ne pas se faire abuser le quota email.

## 4. Risques d'anti-pattern et points de vigilance

1. **RGPD — saut de posture majeur (le plus structurant).** 9.6 stockerait
   `user_email` + **URLs d'annonces suivies** + **historique de prix** = donnees
   personnelles **et** profilage leger (on sait quels biens un email precis surveille,
   son intention d'achat dans la duree), avec **envoi recurrent** non sollicite a
   chaque changement. Cela impose : consentement explicite, **desinscription**
   (lien dans chaque mail), politique de **retention**, **registre de traitement**,
   information claire. C'est un **cran beaucoup plus engageant** que tout ce qui a
   ete fait : la table `Feedback` a ete volontairement concue **en minimisation**
   ("ni IP, ni identifiant, ni extrait d'annonce", `db/models.py:54-58`), et 9.2 a
   recommande de **ne meme pas stocker** l'email (D1). 9.6 va a l'oppose : stockage
   nominatif durable + envoi recurrent. Gros ecart a assumer explicitement.

2. **Cumul des deux prerequis bloquants** (§3) : auth + email. Aucun autre
   requirement 9.x ne cumule les deux. C'est ce qui fait de 9.6 le plus lourd.

3. **Cout / charge — distinguer hash (cheap) et re-LLM (cher).** Si on re-analyse au
   LLM chaque annonce chaque jour : cout = ~0,001 EUR x N annonces x 30 jours +
   latence + risque fallback (`llm_semantic._FALLBACK`). A 100 annonces suivies =
   ~3 EUR/mois, soit potentiellement **plus que tout le cout fixe actuel** (<1
   EUR/mois, CONTEXT §3.2). Or **un simple hash de contenu detecte un changement
   sans LLM**. Le requirement confond "detecter un changement" et "re-analyser".
   Reco : ne re-analyser au LLM **que** si un changement est detecte par hash (et
   meme alors, est-ce utile ?). A arbitrer (Q6).

4. **Anti-bot / promesse intenable.** `url_fetch` echoue sur les grands portails
   (Leboncoin, SeLoger, Bien'ici en HTML : pages JS-only / 403, CONTEXT §4.3 ;
   confirme par le choix d'attaquer Bien'ici via son **API JSON** dans les
   scrapers, CLAUDE.md §8). Or une watchlist d'annonces grand public serait
   massivement composee d'URLs de ces portails -> `fetch_listing_text` renverrait
   `None` -> aucune detection possible -> **promesse produit non tenue** pour la
   majorite des annonces. Quelle couverture reelle promet-on ? (Q5)

5. **Detection "baisse de prix" non triviale.** Le hash detecte **qu'un contenu a
   change**, pas **que le prix a baisse** (un changement de description, d'horaire de
   visite, d'un script analytics injecte declenche aussi le hash). Extraire le prix
   de facon fiable sur une page arbitraire = soit LLM (cher, recurrent, faillible),
   soit parseur par site (inexistant pour une URL libre). Le mail "a baisse de X
   EUR" promet une precision que la techno ne garantit pas. Risque de **fausses
   notifications** (erode la confiance). (Q6)

6. **Faux positifs de hash.** Beaucoup de pages changent a chaque chargement
   (timestamps, tokens CSRF, pubs, contenu dynamique). Un hash brut du texte
   `url_fetch` (deja nettoye de scripts/nav) reduit le bruit mais ne l'elimine pas.
   A prevoir : normalisation, voire ne hasher que la zone prix (qui n'est pas
   isolable sur une page arbitraire — cf. point 5).

7. **Relais de spam (double).** Comme 9.2 §4.2 : un endpoint qui envoie un mail a
   une adresse fournie (magic link) **et** des envois recurrents declenches par
   cron sont deux vecteurs d'abus / de gachis de quota / de blacklistage du domaine
   tout neuf. **Aucun rate limit n'existe.** Garde-fous indispensables.

8. **Nouveau vendor.** Selon les arbitrages : provider email (9.2), eventuellement
   service d'auth tiers (Clerk/Supabase), eventuellement stockage externe.
   Le projet est volontairement minimal (CONTEXT §11.5, pas de dependance non
   justifiee). Chaque vendor = secret, point de panne, surface RGPD.

9. **Redistribution d'annonces (anti-pattern CONTEXT §11.3).** Stocker l'URL et un
   `last_price` par utilisateur reste un usage prive (pas de redistribution
   d'annonces brutes a des tiers ni d'agregation publique) : a priori conforme **si**
   on ne stocke ni ne re-sert le **contenu** de l'annonce a d'autres. A confirmer au
   design : ne stocker que `url` + `last_hash` + `last_price`, pas le texte brut.

10. **Auto-stop Fly** : un cron quotidien reveille la machine via auto_start, comme
    `collect.yml`. **Acceptable**, non bloquant (cout marginal nul, CONTEXT §3.2).

## 5. OPTIONS chiffrees (choix structurants a arbitrer)

### Option AUTH — comment authentifier l'utilisateur (prerequis 1)
- **A1 Magic link maison** (suggere par le requirement) : table tokens + 2 routes +
  session + envoi email. Zero vendor d'auth, mais depend de la brique email (9.2) et
  est un chantier de securite non trivial (TTL, usage unique, rate limit,
  anti-enumeration). Effort reel : plusieurs heures, transversal.
- **A2 OAuth Google/GitHub** : pas d'email a envoyer pour se connecter (contourne en
  partie le prerequis email **pour l'auth**, pas pour la notification), mais nouveau
  vendor + flux OAuth + redirection + toujours une session a gerer. Donnees perso
  Google.
- **A3 Service d'auth tiers** (Clerk, Supabase Auth, Auth0) : rapide a integrer,
  mais nouveau vendor + nouveau stockage de donnees perso hors UE potentiellement +
  cout au-dela du free tier + dependance structurante.
- **A4 Pas de compte du tout** (degrade, cf. §6) : la watchlist vit cote client
  (localStorage), pas d'auth, pas d'email d'auth. Supprime le prerequis 1 entierement.
- Reco analyste : si on veut tester la valeur vite, **A4** (degrade). Sinon, **A1**
  une fois la brique email (9.2) en place, en acceptant que c'est un chantier dedie.

### Option DETECTION — hash seul vs re-LLM (cout)
- **D1 Hash de contenu seul** (`fetch_listing_text` -> `sha256`) : detecte un
  changement, cout quasi nul, pas de LLM. Ne dit pas que c'est le prix.
- **D2 Hash + extraction de prix legere** (regex/heuristique sur le texte) :
  meilleure pour "baisse de prix" mais fragile (formats de prix varies, "FAI",
  "a partir de" ; cf. `scrapers/base.normalize_price` qui existe deja mais cale pour
  des structures connues, pas une page arbitraire).
- **D3 Re-analyse LLM complete** a chaque changement : extrait `price_total`
  proprement (via `run_full_analysis`) mais cout recurrent + latence + risque
  fallback. A reserver au "changement detecte", jamais en balayage quotidien
  systematique.
- Reco analyste : **D1 pour detecter le changement**, et **n'envisager D2/D3 que sur
  changement detecte**, en assumant l'imprecision "le contenu de cette annonce a
  change" plutot que de promettre "a baisse de X EUR". A arbitrer (Q6).

### Option NOTIF — canal de notification
- **N1 Email** (suggere) : depend de la brique email 9.2 (bloquee). Recurrent ->
  desinscription RGPD obligatoire.
- **N2 In-app uniquement** (pastille "cette annonce a change" quand l'utilisateur
  revient) : pas d'email, pas de domaine, mais ne "pousse" pas a revenir (or l'objet
  produit EST de pousser a revenir). Coherent avec une version sans compte (localStorage).
- Reco analyste : **N2** pour un MVP degrade sans email ; **N1** seulement apres
  deblocage 9.2 + auth.

### Option STOCKAGE — ou vit la watchlist
- **S1 SQLite Fly** (modele `Feedback`/`Comparable`, mono-writer, volume `/data`) :
  coherent avec l'archi, zero vendor, mais necessite l'auth pour rattacher les
  lignes a un utilisateur.
- **S2 localStorage client** (degrade) : zero backend, zero RGPD, zero auth ; mais
  pas de cron serveur possible (le navigateur doit etre ouvert) -> la re-analyse
  devient **manuelle** (bouton "re-verifier mes annonces") au lieu d'un push quotidien.
- **S3 Supabase / Postgres externe** : sur-dimensionne pour un MVP <1 EUR/mois +
  nouveau vendor + donnees perso hebergees ailleurs. A exclure pour ce volume.
- Reco analyste : **S1** pour la version complete (cohere avec le repo), **S2** pour
  la version minimale sans compte.

## 6. Challenge du requirement (posture adversariale, sans concession)

- **C'est le chantier le plus lourd de la liste 9.x.** Il cumule auth (inexistante),
  email (deja bloque/differe en 9.2), RGPD recurrent (saut de posture majeur vs la
  minimisation assumee jusqu'ici), fetch fiable (impossible sur les grands portails),
  et detection de prix (non triviale). Aucun autre requirement n'empile autant de
  prerequis non satisfaits.
- **Pour une hypothese non validee.** L'objectif (engagement regulier) repose sur
  les hypotheses d'acquisition explicitement "non validees" (CONTEXT §2.4), sur un
  **MVP gratuit en phase de test** (CONTEXT §3.1). Construire de l'auth + de l'email
  + du profilage RGPD avant d'avoir prouve que les gens reviennent est un pari
  asymetrique : cout (technique, securite, conformite) eleve, gain non prouve.
- **L'estimation "4-6h" est irrealiste** : elle ignore l'auth (chantier entier),
  l'acquisition de domaine (action humaine, deja identifiee comme bloquante en 9.2),
  le rate limit (inexistant), et la conformite RGPD d'un envoi recurrent. Le cout
  reel est un multiple, dont une partie non codable.
- **Version minimale qui apporte de la valeur SANS auth ni email** (a privilegier
  pour tester l'hypothese a moindre risque) :
  - **Degrade D-localStorage** : l'utilisateur garde ses liens d'annonces dans le
    navigateur (localStorage), avec un bouton "re-verifier" qui relance l'analyse
    (`/analyze` existant) a la demande. Zero backend nouveau, zero RGPD, zero auth,
    zero email. Teste le besoin "suivre des annonces" sans aucun des prerequis lourds.
  - **Degrade D-comparateur** : un "avant/apres" sans compte — l'utilisateur colle
    deux versions (ou re-soumet l'URL) et on montre ce qui a change (et le prix s'il
    est extractible). Zero compte, zero email.
  - Ces degrades **suppriment integralement** les deux prerequis bloquants et le
    saut RGPD, tout en validant l'appetence avant d'investir dans l'auth + l'email.
- **Sinon : reporter franchement.** Tant que 9.2 (email/domaine) n'est pas debloque
  et qu'aucune brique d'auth n'existe, le 9.6 "complet" (cron serveur + email push)
  n'est ni livrable ni testable de bout en bout. Le reporter explicitement (comme
  9.3, 9.5, 9.8 le sont deja dans CONTEXT §8) est la position honnete.

## 7. QUESTIONS POUR L'HUMAIN (GATE 1)

> [ACTION EXTERNE] = l'humain doit agir hors code (domaine, DNS, secret, compte
> vendor), bloquant pour la prod. [ARBITRAGE] = decision de conception,
> implementable ensuite par un agent.

**Q0 — [ARBITRAGE, prealable] Faut-il faire 9.6 maintenant, ou tester d'abord
l'hypothese via une version minimale sans compte ni email ?**
9.6 complet cumule deux prerequis bloquants (auth + email) + un saut RGPD, pour une
hypothese d'engagement non validee (§6).
- Reco : **NE PAS faire le 9.6 complet maintenant.** Soit livrer un **degrade
  localStorage** ("mes annonces" cote client + bouton "re-verifier" qui rejoue
  `/analyze") qui teste l'appetence a cout/risque nul ; soit **reporter
  explicitement** 9.6 (comme 9.3/9.5/9.8) tant que 9.2 et une brique d'auth
  n'existent pas. Si l'humain veut malgre tout le 9.6 complet, traiter Q1->Q8.

**Q1 — [ACTION EXTERNE, BLOQUANT PROD] Debloquer la brique email (prerequis partage
avec 9.2, requis DEUX fois ici : magic link + notification).**
Acquerir un domaine, creer le compte provider (Resend/…), verifier le domaine
(SPF/DKIM), poser `RESEND_API_KEY` sur Fly. Sans cela, ni le magic link ni la
notification ne fonctionnent (cf. 9.2 §3.1).
- Reco : prerequis absolu. Tant qu'il n'est pas fait, 9.6 n'est pas livrable.

**Q2 — [ARBITRAGE] Detection : hash seul (D1), hash + extraction prix (D2), ou
re-analyse LLM (D3) ? Et le mail promet-il "le contenu a change" ou "a baisse de X
EUR" ?**
- Reco : **D1 pour detecter le changement** (cout nul, pas de LLM en balayage
  quotidien) ; n'envisager une extraction de prix que **sur changement detecte** ;
  formuler prudemment ("cette annonce a change") plutot que promettre un montant de
  baisse que la techno ne garantit pas (§4.5).

**Q3 — [ARBITRAGE + ACTION EXTERNE selon choix] Mecanisme d'auth utilisateur :
magic link maison (A1), OAuth (A2), service tiers (A3), ou pas de compte (A4) ?**
Aucune auth n'existe (§3.1). Magic link maison = chantier entier + depend de
l'email. Vendor d'auth = nouvelle dependance + donnees perso.
- Reco : **A4 (pas de compte)** pour la version minimale ; sinon **A1** apres
  deblocage email, en l'assumant comme un chantier dedie (pas "inclus dans 4-6h").

**Q4 — [ARBITRAGE] RGPD : assume-t-on le passage d'une posture de minimisation
(Feedback sans email, 9.2 sans stockage) a un stockage nominatif durable + envoi
recurrent + profilage leger ?**
Implique consentement explicite, lien de desinscription dans chaque mail, retention
definie, registre de traitement (§4.1).
- Reco : ne le faire que si la valeur est validee (cf. Q0). Si oui, exiger des le
  design : consentement explicite, desinscription, retention bornee, pas de stockage
  du contenu brut de l'annonce.

**Q5 — [ARBITRAGE] Couverture promise : que fait-on des URLs des grands portails
(Leboncoin/SeLoger/Bien'ici HTML) que `url_fetch` ne peut pas recuperer (§4.4) ?**
- Reco : ne pas promettre le suivi universel. Soit restreindre explicitement aux
  sites fetchables (petites agences, comme idemmo.fr), soit afficher un message
  clair "ce site ne peut pas etre suivi automatiquement". Ne pas laisser une
  watchlist silencieusement morte.

**Q6 — [ARBITRAGE] Re-analyse LLM : balayage quotidien systematique (cout x N
annonces x 30j) ou uniquement sur changement detecte par hash ?**
- Reco : **uniquement sur changement detecte** (et meme la, evaluer si le LLM est
  necessaire). Eviter tout re-appel LLM en balayage quotidien (pourrait depasser le
  cout fixe actuel <1 EUR/mois, §4.3).

**Q7 — [ARBITRAGE] Stockage : SQLite Fly (S1) ou localStorage client (S2) ?**
- Reco : **S2** pour la version minimale (pas d'auth/RGPD, mais re-verification
  manuelle, pas de push) ; **S1** pour la version complete (cohere avec le repo,
  mais necessite l'auth Q3).

**Q8 — [ARBITRAGE] Anti-abus : quel garde-fou minimal (rate limit inexistant
aujourd'hui) sur la demande de magic link et sur les envois recurrents ?**
- Reco : rate limit par IP + plafond global/jour aligne sur le quota du provider,
  anti-enumeration d'emails, lien de desinscription. Prerequis si Q1/Q3 retenus.
