# 9.7 — SPEC : Collecte de feedback utilisateur post-analyse

Role : spec-writer. Cahier des charges implementable. Source : `CONTEXT.md` §9.7,
`docs/specs/9.7-ANALYSE.md`, arbitrages GATE 1 (2026-06-04).
Aucune ligne de solution n'est codee ici.

## 1. Objectif et perimetre

### Objectif
Apres l'affichage d'un resultat d'analyse, proposer un micro-formulaire (note 1-5
+ commentaire optionnel) dont la finalite est **interne** : ameliorer le prompt
LLM et le wording UX. Aucune fonctionnalite visible cote utilisateur au-dela d'un
accuse de reception.

### In
- Backend : 1 endpoint `POST /feedback` public, 1 modele Pydantic `FeedbackIn`,
  1 table SQLite `feedback`.
- Frontend : 1 fonction client `sendFeedback` dans `frontend/lib/api.ts`,
  1 micro-composant de formulaire dans la zone resultat de `frontend/app/page.tsx`,
  generation cote client d'un `analysis_id`.
- Footer : correction de la mention « Aucune donnee conservee ».
- Tests : `backend/tests/test_feedback.py`.

### Out (non-objectifs, voir §5)
- Supabase / Postgres / nouveau vendor.
- Rate-limiting via dependance externe (slowapi).
- A/B testing de prompts (9.8) : seule la colonne `prompt_variant` est pre-cablee,
  sans logique.
- Auth / compte / identification utilisateur.
- Endpoint de lecture/consultation des feedbacks (`GET /admin/feedback`).
- Stockage d'IP, d'identifiant utilisateur, ou d'extrait d'annonce.
- Toute modification du schema `AnalyzeResponse` ou du contrat `/analyze` dans
  `frontend/lib/api.ts`.

## 2. Decisions actees (GATE 1)

1. **Stockage** : OPTION A. Nouvelle table SQLite `feedback` dans le fichier
   `DATABASE_PATH` existant (volume Fly `/data`). PAS de Supabase. Supabase reste
   une piste d'optimisation future, hors perimetre.
2. **Modele de donnees** : D2. Champs `rating` (1-5), `comment` (texte optionnel
   borne), `analysis_id`, `global_score`, `verdict` (au moment du feedback),
   `created_at`. SANS extrait d'annonce. Ajout d'une colonne `prompt_variant`
   (String, nullable) pour pre-cabler 9.8, SANS logique A/B.
3. **Anti-abus** : niveau (b). Garde-fous legers sans nouvelle dependance :
   `rating` 1-5 obligatoire et valide, `comment` borne a 1000 caracteres, rejet des
   corps vides/invalides (422). PAS de slowapi.
4. **RGPD / promesse affichee** : (a)+(c). Corriger la mention « Aucune donnee
   conservee » du Footer pour rester honnete, afficher une micro-mention « Ne
   saisissez pas de donnees personnelles » sous le champ libre, ne stocker NI IP NI
   identifiant utilisateur.
5. **Endpoint** : `POST /feedback` separe et public. On ne touche NI a
   `AnalyzeResponse` NI au contrat `/analyze` de `lib/api.ts`. L'`analysis_id` est
   genere **cote frontend** (`crypto.randomUUID()`) au moment de l'affichage du
   resultat, puis envoye avec le feedback. Ce choix evite de modifier le contrat
   `/analyze` (invariant `lessons.md` ligne 19).

### Points non encore tranches (a remonter si besoin avant dev)
- Retention / purge des feedbacks : aucune politique d'effacement automatique
  decidee. Pour le MVP, on s'appuie sur la minimisation (pas d'IP, pas
  d'identifiant, micro-mention) et la non-incitation a saisir des donnees perso.
  Si une retention bornee est exigee, c'est un ticket separe.

## 3. Contrat technique

### 3.1 Table `feedback` (`backend/db/models.py`)

Nouvelle classe `Feedback(Base)`, `__tablename__ = "feedback"`, declaree a cote de
`Comparable` sur le `Base` partage. Creee par `Base.metadata.create_all` dans
`init_db` (pas d'ALTER : nouvelle table).

| Colonne | Type SQLAlchemy | Nullable | Contraintes / notes |
|---|---|---|---|
| `id` | `Integer` | non | `primary_key=True, autoincrement` |
| `rating` | `Integer` | non | valeur attendue 1-5 (validation Pydantic, voir 3.2) |
| `comment` | `String` | oui | longueur max 1000 (validation Pydantic) |
| `analysis_id` | `String` | oui | UUID genere cote front, opaque cote backend |
| `global_score` | `Integer` | oui | score affiche au moment du feedback |
| `verdict` | `String` | oui | verdict affiche au moment du feedback |
| `prompt_variant` | `String` | oui | pre-cablage 9.8, aucune logique en 9.7 |
| `created_at` | `DateTime` | non | `default=datetime.utcnow` |

Aucune colonne d'adresse IP, d'identifiant utilisateur, de cookie ou d'extrait
d'annonce. Interdiction explicite (anti-pattern RGPD, §6).

### 3.2 Modele Pydantic `FeedbackIn` (`backend/app/main.py`)

Champs :
- `rating: int` — **obligatoire**, valide dans `[1, 5]` (bornes incluses). Hors
  bornes ou absent -> 422.
- `comment: Optional[str] = None` — longueur max 1000 caracteres. Au-dela -> 422.
- `analysis_id: Optional[str] = None`.
- `global_score: Optional[int] = None`.
- `verdict: Optional[str] = None`.
- `prompt_variant: Optional[str] = None`.

La validation des bornes (`rating` 1-5, `comment` <= 1000) doit etre portee par le
modele Pydantic (contraintes de champ), de sorte qu'une violation produise un 422
FastAPI standard, sans code de validation manuel dans la route.

### 3.3 Endpoint `POST /feedback` (`backend/app/main.py`)

- Methode/chemin : `POST /feedback`. **Public** (aucun `X-Admin-Token`), aligne sur
  `/analyze`.
- Body : `FeedbackIn` (JSON). Corps vide ou JSON sans `rating` -> 422 (validation
  Pydantic).
- Comportement : persiste une ligne `Feedback` via `SessionLocal()` (ouverture /
  `commit` / `close` dans un `try/finally`, calque sur les routes existantes).
- Logging : logger nomme `feedback` (`logging.getLogger("feedback")`), niveau INFO.
  Logger l'enregistrement (ex. `rating`, presence de commentaire, `analysis_id`)
  SANS jamais logger le contenu du commentaire (peut contenir des donnees perso).
- Reponse succes : code **201 Created**, corps JSON `{"status": "ok"}`.
- Ne lit, ne resout, ni ne stocke l'adresse IP de l'appelant.

### 3.4 Migration / init DB (`backend/db/session.py`)

- Aucune entree a ajouter dans `_ADD_COLUMNS` : `feedback` est une table neuve,
  `create_all` la cree integralement.
- Le hook `@app.on_event("startup")` -> `init_db()` reste inchange (ne pas
  refactorer vers lifespan dans ce ticket).

### 3.5 Variables d'environnement / secrets

Aucune nouvelle variable, aucun nouveau secret. La table cohabite dans le fichier
designe par `DATABASE_PATH` existant.

### 3.6 Contrat frontend

- `frontend/lib/api.ts` : ajouter une fonction `sendFeedback(payload)` calquee sur
  `analyzeListing` (fetch `POST` vers `NEXT_PUBLIC_API_URL + "/feedback"`, header
  `Content-Type: application/json`, gestion d'erreur non bloquante pour l'UX).
  Le payload porte `rating`, `comment` (optionnel), `analysis_id`, `global_score`,
  `verdict`. **Ne pas** modifier `analyzeListing`, `ApiResult`, ni `ApiPillar`.
- `frontend/app/page.tsx` :
  - generer un `analysis_id` (`crypto.randomUUID()`) au passage en etat `result`
    (et a chaque re-analyse via `handleRefine`), le conserver en etat local ;
  - ajouter un etat local `feedbackSent` (accuse de reception apres envoi) ;
  - greffer le micro-formulaire dans le bloc `appState === "result"` (apres
    `LeversList`), envoyer `global_score` et `verdict` issus de `result`.
- `frontend/components/design/Footer.tsx` : remplacer le libelle « Aucune donnee
  conservee » par une mention honnete et coherente avec le stockage d'un feedback
  (le wording exact est laisse au dev, mais il ne doit plus affirmer qu'aucune
  donnee n'est conservee).
- Micro-mention sous le champ libre : « Ne saisissez pas de donnees personnelles ».

Le formulaire reutilise le design system existant (tokens `--ink`, `--parchment`,
`--stone-line`, polices `--font-serif`/`--font-sans`). Pas de nouvelle lib UI.

## 4. Criteres d'acceptation (observables, testables par pytest)

Les tests ciblent `POST /feedback` via le `TestClient` (`conftest.py`, base SQLite
jetable). Numerotation contractuelle pour testeur et reviewer.

1. **Golden path** : `POST /feedback` avec `{"rating": 5, "comment": "Tres clair",
   "analysis_id": "<uuid>", "global_score": 82, "verdict": "Coherence forte"}`
   renvoie **201**. Corps de reponse JSON contenant `{"status": "ok"}`.
2. **Rating minimal sans commentaire** : `POST /feedback` avec `{"rating": 3}`
   (sans `comment` ni autres champs) renvoie **201** (les champs D2 sont optionnels).
3. **Rating manquant** : `POST /feedback` avec un corps `{"comment": "test"}` (pas
   de `rating`) renvoie **422**.
4. **Rating hors bornes (haut)** : `POST /feedback` avec `{"rating": 6}` renvoie
   **422**.
5. **Rating hors bornes (bas / zero)** : `POST /feedback` avec `{"rating": 0}`
   renvoie **422**.
6. **Rating de type invalide** : `POST /feedback` avec `{"rating": "bien"}` renvoie
   **422**.
7. **Commentaire trop long** : `POST /feedback` avec `{"rating": 4, "comment": "x"
   * 1001}` renvoie **422** (rejet, PAS de troncature).
8. **Corps vide** : `POST /feedback` avec un corps JSON `{}` renvoie **422**.
9. **Persistance verifiable** : apres un `POST /feedback` valide (201), une requete
   sur la table (`SessionLocal().query(Feedback).count()` ou egal) montre
   exactement une ligne de plus, avec `rating`, `comment`, `analysis_id`,
   `global_score`, `verdict` egaux aux valeurs envoyees et `created_at` non nul.
10. **Aucune IP stockee** : le modele `Feedback` n'expose aucune colonne
    d'adresse IP ni d'identifiant utilisateur. Verifiable par introspection
    (`PRAGMA table_info(feedback)` ou `Feedback.__table__.columns`) : l'ensemble
    des colonnes est exactement `{id, rating, comment, analysis_id, global_score,
    verdict, prompt_variant, created_at}`, sans `ip`, `client_ip`, `user_id`,
    `session_id` ni equivalent.
11. **Endpoint public** : `POST /feedback` valide aboutit en 201 **sans** header
    `X-Admin-Token` (l'ecriture n'est pas protegee par token).
12. **Contrat `/analyze` intact** : le schema de reponse de `POST /analyze`
    (`global_score`, `verdict`, `confidence`, `pillars[]`, `actions{check,
    questions, negotiation}`) est inchange ; aucun champ ajoute/retire.
    `AnalyzeRequest`/`AnalyzeResponse` non modifies.

## 5. Hors-perimetre / non-objectifs

- Supabase / Postgres / tout nouveau vendor ou secret.
- Rate-limiting via slowapi ou autre dependance ; pas de protection anti-spam
  au-dela de la validation Pydantic.
- A/B testing de prompts et toute lecture/exploitation de `prompt_variant`.
- Endpoint d'administration de lecture des feedbacks.
- Auth, compte, cookies, tracking comportemental.
- Politique de retention / purge automatique des feedbacks.
- Migration `on_event` -> lifespan, refactor de `init_db`.
- Stockage d'IP, identifiant, ou extrait d'annonce.

## 6. Conformite (anti-patterns applicables)

- **RGPD / minimisation** : ne stocker NI IP NI identifiant utilisateur NI extrait
  d'annonce. Le commentaire libre n'est jamais journalise en clair. Micro-mention
  « Ne saisissez pas de donnees personnelles » obligatoire sous le champ libre
  (invariant consentement, `lessons.md` ligne 23).
- **Coherence de la promesse affichee** : le Footer ne doit plus affirmer « Aucune
  donnee conservee » des lors qu'un feedback est persiste.
- **Pas de secret en clair** : aucun secret introduit ; rien a committer.
- **Contrat `/analyze` stable** : ne pas modifier `AnalyzeResponse` ni
  `frontend/lib/api.ts` cote `/analyze` (`lessons.md` ligne 19). Le feedback est un
  endpoint distinct ; l'`analysis_id` est genere cote front.
- **Pas d'estimation de prix, pas de redistribution d'annonces brutes** : non
  concerne par ce ticket, mais aucune donnee d'annonce brute n'est stockee.
- **Conventions CLAUDE §12** : Python 3.12, logger nomme `feedback`, pas de
  commentaire « what », pas d'emoji.
