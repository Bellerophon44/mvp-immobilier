## 0. Reformulation et périmètre

### Objectif
Remplacer le « 100 % hypothèses » du FORECAST (trafic M0 inconnu, H5, item #1 du
§10 « CRITIQUE ») par des **faits mesurés**, et **protéger** `/analyze` contre
l'abus (coût LLM + disponibilité). Le FORECAST classe ces deux briques #1
(prérequis) et #2 (garde-fou) de tout le backlog (§6.1).

### In (ce qui est demandé)
- A1. Table SQLite `events` (esprit `Feedback` : pas d'IP, pas de PII, bands /
  agrégats seulement).
- A2. Endpoint `POST /events` anonyme (public, pattern `/feedback`).
- A3. Émission front des events de la liste FORECAST §7.1, en **fire-and-forget**
  (ne réveille pas la VM par polling, ne casse pas le contrat `/analyze`).
- B1. Rate-limit en mémoire sur `POST /analyze` (IP transitoire, jamais
  persistée), réponse 429 au-delà du seuil.

### Out (hors périmètre, à acter)
- Tout SaaS analytics (Plausible / Umami / GA). Le périmètre proposé est
  **first-party** : c'est cohérent avec « < 1 €/mois », « pas de nouveau vendor »,
  « pas de secret externe ». À confirmer en Q1.
- Dashboards / rapports d'exploitation des events (lecture, corrélations). Le
  FORECAST §7.2 les décrit mais c'est un livrable distinct (lecture, pas collecte).
  Même découpage que 9.7 (« rapports d'exploitation » notés en suite, non livrés).
- Auth, comptes, cookies de session identifiants (bloqués / hors sujet, cf. §9.6).
- Rate-limit sur `/feedback` et `/events` eux-mêmes (à discuter, Q6 — pas dans le
  libellé mais c'est le corollaire honnête de B1 : ouvrir un nouvel endpoint
  d'écriture sans limite recrée le risque qu'on ferme côté `/analyze`).

## 1. Confrontation au code réel

### 1.1 `POST /events` — où ça s'insère (`backend/app/main.py`)

Le gabarit existe déjà à l'identique : `POST /feedback` (main.py:308-332) est
public, status 201, écrit via `SessionLocal()` dans un `try/finally`, et
**ne journalise jamais** le champ libre. `/events` se calque dessus :
- Modèle Pydantic `EventIn` à côté de `FeedbackIn` (main.py:77-83).
- Handler `@app.post("/events", status_code=201)` à côté de
  `submit_feedback` (main.py:308).
- Import `Event` ajouté à la ligne `from db.models import Comparable, Feedback`
  (main.py:13).
- Logger dédié `events_logger = logging.getLogger("events")` (cf. main.py:26 pour
  `feedback_logger`) — à ajouter à la liste des loggers nommés de
  `backend/CLAUDE.md` §12.

Point d'attention CORS (cf. §3) : `/events` hérite automatiquement du middleware
CORS global (main.py:40-47), donc l'appel cross-origin Vercel → Fly fonctionne
sans config supplémentaire. Rien à ajouter.

### 1.2 Micro-migration idempotente (`backend/db/session.py`)

`init_db()` (session.py:58-61) fait `Base.metadata.create_all` puis une
micro-migration **uniquement sur `comparables`** (`_migrate_comparables`,
session.py:51-55, boucle `_ADD_COLUMNS`). Conséquence concrète :

- Une **nouvelle table** `Event` déclarée comme classe SQLAlchemy dans
  `db/models.py` est créée automatiquement par `create_all` au prochain
  `init_db` (au `@app.on_event("startup")`, main.py:50-52). **Aucune DDL manuelle
  n'est nécessaire pour créer la table** — c'est le même mécanisme que la table
  `feedback` (créée sans entrée dans `_ADD_COLUMNS`).
- `_migrate_comparables` ne gère que l'ajout de **colonnes** à une table
  existante. Il ne faut donc rien y toucher pour 9.9, **sauf** si on prévoit que
  le schéma `events` évoluera ensuite : dans ce cas, prévoir dès le départ un
  `_migrate_events` symétrique (même PRAGMA table_info + ADD COLUMN) pour ne pas
  recréer la dette « create_all ne fait pas d'ALTER ». Recommandation : créer la
  table proprement maintenant, ajouter `_migrate_events` **seulement** si une
  colonne est ajoutée plus tard (YAGNI).

Idempotence : `create_all` est déjà idempotent (ne recrée pas une table
existante). Aucun risque sur la base prod `/data/comparables.db`.

### 1.3 Émission front sans casser `/analyze` ni réveiller la VM

Contraintes vérifiées dans le code :
- `frontend/lib/api.ts` : `sendFeedback` (api.ts:94-108) est déjà un envoi
  **non bloquant** avec `try/catch` qui avale l'erreur (« un echec de feedback ne
  doit jamais degrader l'UX »). Un `sendEvent(name, props)` se calque dessus.
- `frontend/app/page.tsx` est `"use client"` : les hooks d'émission sont
  triviaux à brancher aux points existants —
  - `analysis_started(mode)` : début de `handleAnalyze` (page.tsx:437) ;
  - `analysis_succeeded(...)` / `analysis_failed(reason)` : dans le `try`/`catch`
    de `handleAnalyze` (page.tsx:445-452) — **après** réception de `res`, donc le
    résultat est disponible (band de `res.global_score`, `res.confidence`,
    `res.pillars[0].verdict` pour `pillar_price_status`) ;
  - `report_export(format)` : `handleCopy` (page.tsx:548) et `handlePrint`
    (page.tsx:559) ;
  - `district_refine(from_scope→to_scope)` : `handleRefine` (page.tsx:458) — le
    `from_scope` est `result.pillars[0].scope` **avant** la ré-analyse, le
    `to_scope` celui **après** ;
  - `address_entered(bool)` : `handleAddressSubmit` (page.tsx:480) — booléen
    seulement, **jamais la chaîne adresse** (PII) ;
  - `feedback_submitted` : déjà câblé via `/feedback` (voir §1.4) ;
  - `page_view` / `methode_view` : émis au montage (`useEffect`) de
    `app/page.tsx` et `app/methode/page.tsx`. **Piège** : `methode/page.tsx` est
    aujourd'hui un **server component** (pas de `"use client"`, il exporte
    `metadata`, methode/page.tsx:6-10). Émettre un event y impose soit un petit
    composant client dédié (`<PageViewBeacon name="methode_view" />`) monté dans
    la page, soit de passer la page en client (à éviter : casse l'export
    `metadata` / le SSR SEO). **Reco : composant beacon client minimal**, zéro
    impact sur le rendu serveur.

Fire-and-forget vs réveil VM : l'auto-stop Fly se réveille sur **toute** requête
HTTP entrante, y compris `POST /events`. La nuance « ne pas réveiller la VM »
(leçon §9.3) vise le **polling périodique** (cron / setInterval qui réveille en
boucle une VM idle et fait grimper la facture). Un event émis **uniquement sur
action utilisateur** ne réveille la VM que quand un humain utilise le site —
c'est-à-dire quand `/analyze` allait de toute façon la réveiller. Donc :
- `page_view` réveille la VM **dès l'arrivée d'un visiteur**, avant même
  `/analyze`. C'est un léger surcoût (réveils sur des visiteurs qui ne lancent
  pas d'analyse) mais **borné par le trafic réel** (< 1 €/mois reste tenu :
  réveiller une VM est gratuit, seul le temps CPU coûte, négligeable pour un
  INSERT). À acter quand même (Q5 : `page_view` côté serveur vs côté client-only).
- `keepalive: true` sur le `fetch` (ou `navigator.sendBeacon`) est utile pour les
  events émis **au unload** (ex. un futur `page_close`). Pour les events
  in-session (la liste 9.9), un `fetch` non-awaited suffit ; `keepalive` est une
  ceinture de sécurité bon marché. **Ne pas `await`** ces fetch dans le chemin
  critique de `handleAnalyze` : l'event ne doit jamais retarder ni faire échouer
  l'analyse.

Contrat `/analyze` : **intact**. 9.9 n'ajoute ni champ à `AnalyzeRequest`/
`AnalyzeResponse` ni header. C'est un endpoint séparé + des appels front
additionnels. Anti-pattern §11.9 (changer la signature `/analyze`) **non
déclenché**. C'est le bon design, comme 9.7 l'a fait pour le feedback.

### 1.4 `feedback_submitted` : on le route via `/events` ou pas ?

État réel : le feedback est **déjà** persisté en table `feedback` via
`POST /feedback` (main.py:308, `sendFeedback` api.ts:94). Le router via `/events`
le **dupliquerait**. Trois options :
- (a) **Ne pas** émettre `feedback_submitted` vers `/events` ; lire la table
  `feedback` quand on aura besoin de ce signal (corrélation note ↔ score existe
  déjà dans `feedback`). **Reco** : zéro duplication, zéro divergence de
  comptage. Le FORECAST §7.1 liste `feedback_submitted` comme event « base A/B
  futur » — mais la donnée est déjà collectée ailleurs.
- (b) Émettre un event `feedback_submitted` léger (rating + prompt_variant
  seulement) **en plus**, pour que le funnel events soit complet dans une seule
  table. Risque : deux sources de vérité pour le même fait → écarts de comptage.
- (c) Migrer le feedback dans `events` (refonte). **Hors périmètre**, casserait
  9.7. Écarté.

→ Remonté en Q7. Reco analyste : (a).

### 1.5 Rate-limit en mémoire sur une VM auto-stop mono-instance

Faits :
- 1 seule machine Fly (`auto_stop_machines = true`, `min_machines_running = 0`,
  CONTEXT §0 / CLAUDE §2), SQLite mono-writer → **mono-instance**. Un compteur
  en mémoire process (`dict[ip] -> deque[timestamps]`) est donc **cohérent** :
  pas de second réplica qui aurait son propre compteur. C'est le cas favorable.
- **État perdu au restart / cold start** : à chaque réveil de la VM après
  auto-stop, le process redémarre → le dictionnaire est vide. Même observation
  que le cache LLM (« perdu au restart », CLAUDE §11) et le compteur LLM-failure
  écarté en 9.3 pour cette raison.

Est-ce acceptable ? **Oui, pour l'objectif visé.** Le rate-limit ici protège
contre un **abus en rafale** (boucle qui martèle `/analyze` pour cramer le budget
LLM). Un attaquant qui martèle maintient justement la VM **éveillée** → le
compteur **ne** se réinitialise **pas** pendant l'attaque (l'auto-stop n'arrive
qu'après une période d'inactivité). La fenêtre « état perdu » n'existe que **entre**
deux salves espacées de plusieurs minutes — ce qui n'est plus une rafale. Donc la
limite en mémoire **fait son travail sur le seul scénario qui compte** (rafale).
Elle ne protège pas contre un abus « lent et distribué » (1 req toutes les N
minutes depuis 1000 IP) — mais ce scénario coûte ~0,001 €/analyse (§3.2) et est
de toute façon hors de portée d'un rate-limit par IP en mémoire.

Garde-fou complémentaire **déjà recommandé** et non technique : **usage limit
hard OpenAI** (CONTEXT §9.4, FORECAST §5.2). Le rate-limit et l'usage limit sont
**complémentaires** : le premier protège la dispo + lisse, le second est le
filet financier dur. Le rate-limit **ne remplace pas** l'usage limit (Q8).

Implémentation : **pas de nouvelle dépendance nécessaire.** `slowapi` (cité et
écarté en 9.7 comme sur-dimensionné) ajoute une dep ; une **dépendance FastAPI
maison** de ~20 lignes (dict IP→deque de timestamps, fenêtre glissante, `429`
avec `Retry-After`) suffit et reste dans l'esprit « 0 dépendance » du projet
(requirements.txt n'a que 7 lignes). Reco : maison. À confirmer Q4.

IP : récupérée via `request.client.host` **ou**, derrière le proxy Fly, l'en-tête
`Fly-Client-IP` / `X-Forwarded-For` (Fly place l'IP réelle dans `Fly-Client-IP`).
**Piège** : `request.client.host` derrière Fly peut renvoyer une IP interne du
proxy → tout le monde partagerait le même bucket (rate-limit global au lieu de
par-IP). Il faut lire `Fly-Client-IP` en priorité. **Cette IP n'est ni
journalisée ni persistée** — elle vit uniquement comme clé de dict en mémoire,
purgée par fenêtre glissante (cf. RGPD §2).

## 2. RGPD — qu'est-ce qui pourrait devenir une PII ?

Le projet a une posture de **minimisation assumée** (modèle `Feedback` : « ni IP,
ni identifiant, ni extrait d'annonce », models.py:54-70). 9.9 doit la tenir.
Points de vigilance, du plus au moins risqué :

1. **`referrer` complet** (FORECAST §7.1 précise déjà « domaine seul »). Un
   referrer brut peut contenir des paramètres d'URL identifiants (UTM, tokens,
   parfois un email en query string sur des liens mal faits). **Garde-fou : ne
   stocker que le domaine** (`new URL(document.referrer).hostname`), jamais le
   path ni la query. À tester (verrou type 9.7).
2. **L'IP du rate-limit (B)**. Une IP est une **donnée personnelle** au sens
   RGPD. Tant qu'elle reste une **clé de dict en mémoire transitoire, jamais
   journalisée, jamais écrite sur disque**, et purgée en continu, le traitement
   est minimal et proportionné (sécurité du service = intérêt légitime). **La
   ligne rouge** : ne jamais l'écrire dans la table `events`, ni dans un log, ni
   dans la DB. Le libellé du requirement l'impose explicitement (« SANS la
   persister ») — c'est un invariant à verrouiller par revue + test.
3. **Empreinte de session anonyme** (pour `repeat_session` / proxy rétention,
   FORECAST §7.1 « cookie 1ʳᵉ partie ou empreinte douce non-PII »). C'est le
   point le **plus délicat**. Deux familles :
   - **Cookie/`localStorage` 1ʳᵉ partie** contenant un UUID aléatoire (pas dérivé
     de l'IP/UA) : techniquement un identifiant, mais **non-PII** (ne désigne pas
     une personne, juste un navigateur) et **1ʳᵉ partie non publicitaire**. La
     CNIL exempte de consentement les cookies de **mesure d'audience** strictement
     nécessaires si configurés en conformité (pas de croisement, pas de suivi
     inter-sites, finalité limitée). Mais ce **n'est pas** « zéro cookie ».
   - **Empreinte (fingerprint) douce** (hash UA + langue + …) : **à éviter**, la
     CNIL la traite comme un traceur soumis à consentement et c'est plus intrusif
     qu'un cookie 1ʳᵉ partie. Écarter.
   - **Aucun identifiant** : pas de `repeat_session` fiable ni de
     `analysis_volume_per_session` cross-page. On garde un compteur **en mémoire
     d'onglet** (variable JS de la session de navigation, perdu au refresh) →
     proxy très faible mais **zéro cookie / zéro consentement**.
   → Arbitrage structurant remonté en Q3. C'est **le** point qui peut faire
   basculer le besoin de bandeau (Q2).
4. **Affirmations UI existantes à ne pas contredire**. Deux endroits affirment
   au user que rien n'est conservé :
   - `frontend/components/design/Footer.tsx:26` : « Analyses non conservées ».
   - `frontend/app/methode/page.tsx:154` : « Vos analyses ne sont pas conservées. »
   Ces phrases visent le **contenu d'annonce** (vrai : `/analyze` ne persiste
   rien). Instrumenter des **events anonymes agrégés** ne contredit pas
   stricto sensu « analyses non conservées » (on ne stocke pas l'annonce), mais
   c'est **limite** et un utilisateur attentif pourrait l'opposer. Si on pose un
   cookie de session (Q3 option cookie), il faudra **mettre à jour le wording**
   (micro-mention « mesure d'audience anonyme », comme la micro-mention RGPD
   ajoutée en 9.7). Sans cookie et avec events anonymes agrégés, le wording
   actuel reste défendable. → lié à Q2/Q3.
5. **`global_score` en band, pas en valeur exacte.** Le FORECAST §7.1 dit « band »
   pour `analysis_succeeded`. Une valeur exacte n'est pas une PII, mais rester en
   bands (ex. `<40 / 40-59 / 60-79 / 80+`) est cohérent avec la minimisation et
   suffit au funnel. Reco : bands.

**Consentement / bandeau** : si la collecte est **anonyme, agrégée, first-party,
sans cookie identifiant** (events purs, IP non persistée), la doctrine CNIL
n'impose **pas** de consentement ni de bandeau (mesure d'audience exemptée /
absence de traceur). Si on pose un **cookie de session** (Q3), il faut vérifier
qu'il rentre dans l'exemption « mesure d'audience » CNIL (config conforme) pour
rester sans bandeau ; sinon bandeau requis. **À confirmer (Q2)** — l'analyste
recommande la voie sans cookie pour éviter d'ouvrir ce débat sur un MVP.

## 3. Pièges techniques

- **CORS `/events`** : aucun travail — le middleware global (main.py:40-47)
  couvre `*.vercel.app` + localhost et `allow_methods=["*"]`. Vérifié.
- **Coût d'écriture SQLite mono-writer** : `page_view` est l'event le plus
  volumineux (1 par visiteur). SQLite encaisse facilement le volume du FORECAST
  (base M12 ~700 visiteurs/mois, ambitieux ~3000/mois = quelques dizaines
  d'écritures/jour). **Aucun problème de volume** à l'horizon du modèle. Le
  mono-writer ne devient un sujet que « > quelques req/s » (FORECAST §5.3) — très
  loin. Garde-fou tout de même : le `POST /events` doit ouvrir/fermer sa session
  vite (pattern `/feedback`, try/finally) et ne **jamais** bloquer si l'écriture
  échoue (avaler l'erreur côté serveur, renvoyer 201 quoi qu'il arrive ? — ou
  201/500 mais front fire-and-forget de toute façon ; reco : ne pas faire
  échouer le front, c'est déjà le cas par construction côté client).
- **Ordre des events vs disponibilité du résultat** : `analysis_succeeded` doit
  être émis **après** `setResult(res)` (page.tsx:446) pour disposer des bands.
  `analysis_failed(reason)` distingue : `no_input` (400, jamais atteint côté
  front car l'UI empêche l'envoi vide), `url_unreachable` (422, message backend),
  `llm_fallback` — **problème** : le fallback LLM est **invisible côté front**.
  `run_full_analysis` renvoie un 200 normal même sur fallback (le `_FALLBACK` est
  servi comme une analyse réussie, cf. 9.3 : émis en un seul point
  `llm_semantic.py`, jamais signalé au client). Donc le front **ne peut pas**
  émettre `analysis_failed(reason=llm_fallback)` de façon fiable. Deux options :
  - (a) accepter que `analysis_failed` ne couvre que `url_unreachable` /
    `no_input` côté front, et mesurer le `llm_fallback` **côté serveur** (le seul
    qui le connaît) — cohérent avec 9.3 qui a déjà verrouillé le marqueur
    `LLM call failed`. C'est un event **serveur**, pas client.
  - (b) faire remonter un flag `degraded` dans `AnalyzeResponse` → **change le
    contrat** (anti-pattern §11.9), à éviter pour 9.9.
  → Reco : (a). Implique que **certains events sont émis côté serveur**
   (`analysis_failed(llm_fallback)`), d'autres côté client. À acter (Q9).
- **Double-comptage du refine** : `handleRefine` (page.tsx:458) rejoue
  `/analyze`. Il ne faut pas que chaque refine compte comme un nouveau
  `analysis_started` (sinon `taux_lancement` est faussé). Reco : `district_refine`
  est son **propre** event ; `analysis_started` n'est émis que dans
  `handleAnalyze`, pas dans `handleRefine`.
- **`prompt_variant`** : listé dans FORECAST §7.1 mais **9.8 est différé** et le
  front ne l'envoie pas encore (CONTEXT §9.8). Ne pas chercher à le câbler ici :
  hors périmètre, dépendance non satisfaite.

## 4. Dépendances et ordre

- **9.9-A (events) ne dépend de rien de bloquant.** Tous les points d'ancrage
  front existent (handlers déjà écrits), la table se crée via `create_all`, CORS
  est déjà bon, le pattern `/feedback` est livré (9.7). Prérequis interne :
  réutiliser `sendFeedback` comme modèle (api.ts:94).
- **9.9-B (rate-limit) ne dépend de rien** et est **indépendant** de A. Il touche
  `/analyze` (middleware ou dépendance FastAPI) + lecture `Fly-Client-IP`.
- **Dépendance externe non bloquante mais recommandée** : usage limit hard OpenAI
  (§9.4, action humaine 5 min) — à faire **avant ou en parallèle** de B, car B
  seul ne plafonne pas la dépense (Q8).
- **Aval** : 9.9-A est **prérequis** de la quasi-totalité des décisions du
  FORECAST (#1, §6.1) — choix B2C/B2B (signal volume/session), justification d'un
  budget paid (mesurer un CAC), priorisation SEO. 9.8 (A/B) consomme aussi le
  signal satisfaction. Aucune de ces suites n'est dans 9.9.
- **Prérequis manquant signalé** : `llm_fallback` comme event nécessite la voie
  **serveur** (cf. §3) ; ne pas le promettre côté client.

## 5. Risques d'anti-pattern (registre CONTEXT §11 / CLAUDE §1)

| Anti-pattern | Risque en 9.9 | Statut |
|---|---|---|
| §11.9 Changer la signature `/analyze` sans MAJ `api.ts` | A et B touchent `/analyze` ? | **Évité** si events = endpoint séparé + rate-limit = middleware/dépendance (ni body ni schéma de réponse modifiés). À verrouiller en revue. |
| RGPD / minimisation (esprit `Feedback`) | referrer brut, IP persistée, cookie/fingerprint | **Risque réel** → §2. Garde-fous : domaine-seul, IP jamais écrite, pas de fingerprint. |
| §11.3 Redistribuer / stocker l'annonce brute | Un event pourrait inclure l'URL ou un extrait | **Risque** : interdire formellement `path` d'URL d'annonce, texte, adresse dans tout event. Bands / booléens / enums seulement. |
| Nouveau vendor / secret | SaaS analytics | **Évité** par le choix first-party (à confirmer Q1). Pas de secret, pas de DNS, pas de domaine (contrairement à 9.2/9.6). |
| Coût / < 1 €/mois | Réveils VM sur `page_view`, volume INSERT | **Maîtrisé** (§3). Borné par le trafic réel, négligeable. |
| Estimation de prix / DVF | — | Sans objet (pas de logique produit touchée). |
| `slowapi` / nouvelle dépendance | rate-limit | **Évitable** : dépendance FastAPI maison (Q4). |
| Cookie sans base légale | `repeat_session` | **Risque** → Q2/Q3. Voie sans cookie recommandée. |

## 6. Séquencement recommandé

**Recommandation : SCINDER en deux requirements livrés dans cet ordre.**

- **9.9 = rate-limit `/analyze` (la brique B), d'abord.**
  - C'est le **plus court** (~0,5-1 j-h, FORECAST §6), **sans surface RGPD
    structurante** (IP transitoire, déjà cadrée par le libellé), **sans question
    ouverte pour l'humain** sauf le seuil (Q4) et le rappel usage-limit (Q8).
  - C'est un **garde-fou** : tant qu'il n'est pas là, `/analyze` est exposé. Le
    livrer en premier sécurise avant d'ouvrir un **nouvel** endpoint d'écriture
    public (`/events`) — qui sans rate-limit recréerait une surface d'abus (Q6).
  - Aucune dépendance, testable seul (test : 200 sous le seuil, 429 au-delà,
    bucket par IP via `Fly-Client-IP`, état non persisté).

- **9.10 = instrumentation events (la brique A), ensuite.**
  - Porte **toutes** les questions structurantes (stockage, session, rétention,
    consentement) → mérite sa propre passe GATE 1 / SPEC une fois Q1-Q3, Q5, Q7
    tranchées.
  - Plus de surface (table + endpoint + N points front + 2 pages + RGPD). La
    découper évite de bloquer le garde-fou B derrière les arbitrages RGPD de A.

> **Alternative (si l'humain préfère un seul lot)** : tout faire en 9.9 combiné.
> Acceptable car les deux briques sont techniquement indépendantes et petites
> (FORECAST : ~1-3 j-h + ~0,5-1 j-h). Inconvénient : le garde-fou rate-limit
> attend que les questions RGPD des events soient tranchées avant d'être mergé.
> **Reco analyste : scinder** (rate-limit ne doit pas attendre).

Numérotation proposée : **9.9 = rate-limit**, **9.10 = events**. (Le libellé
d'origine nomme 9.9 l'instrumentation ; si l'humain tient à garder 9.9 = events,
inverser sans conséquence — l'important est l'**ordre** : garde-fou avant nouvel
endpoint public.)

## 8. QUESTIONS POUR L'HUMAIN (GATE 1)

1. **Stockage des events : first-party table SQLite `events` (reco) vs SaaS
   (Plausible/Umami self-host) ?**
   - Options : (a) table maison `events` esprit `Feedback` — zéro vendor, zéro
     secret, zéro coût, sous notre contrôle RGPD ; (b) SaaS analytics — plus de
     dashboards prêts mais nouveau vendor + (souvent) cookie + secret + coût.
   - **Reco : (a) first-party.** C'est le périmètre proposé et c'est cohérent avec
     « < 1 €/mois / pas de nouveau vendor ».

2. **Faut-il un bandeau / consentement RGPD ?**
   - Options : (a) **non** — events anonymes, agrégés, first-party, **sans cookie
     identifiant** et IP non persistée → exemption mesure d'audience CNIL, pas de
     bandeau ; (b) **oui** — seulement si on pose un cookie de session non
     exempté (dépend de Q3).
   - **Reco : (a)**, à condition de retenir la voie « sans cookie » en Q3. Confirme
     l'absence d'obligation de bandeau pour un MVP.

3. **Gestion de la session anonyme (pour `repeat_session` / volume/session, proxy
   pro-B2B) :**
   - Options : (a) **rien** — compteur en mémoire d'onglet seulement, proxy
     rétention très faible, zéro cookie / zéro consentement ; (b) **cookie /
     localStorage 1ʳᵉ partie** avec UUID aléatoire non-PII — proxy rétention
     fiable mais introduit un identifiant (vérifier exemption CNIL, peut imposer
     une micro-mention et MAJ du wording « non conservées ») ; (c) fingerprint
     douce — **écartée** (traceur soumis à consentement, plus intrusif).
   - **Reco : (a) pour la première version** (zéro friction RGPD), (b) seulement
     si le **signal B2B** (volume/session récurrent) devient un objectif explicite
     et qu'on accepte la micro-mention + MAJ wording. C'est **le** choix qui
     conditionne Q2.

4. **Politique de rate-limit `/analyze` : seuil et réponse.**
   - Options de seuil (par IP, fenêtre glissante) : (a) **10 req / min / IP**
     (confortable pour un humain qui teste, bloque les rafales) ; (b) 20 req/min ;
     (c) plus strict 5 req/min. Réponse : **429** + `Retry-After`. Implémentation :
     dépendance FastAPI **maison** (reco) vs `slowapi` (dépendance).
   - **Reco : (a) 10 req/min/IP, 429 + Retry-After, implémentation maison.** IP
     lue via `Fly-Client-IP`, jamais persistée ni journalisée.

5. **`page_view` : émis côté serveur (réveille la VM à chaque visiteur) ou
   strictement côté client ?**
   - Options : (a) `POST /events` à l'arrivée — donnée fiable mais réveille la VM
     même pour les visiteurs qui ne lancent pas d'analyse (surcoût borné,
     négligeable) ; (b) ne compter que les visiteurs qui interagissent (pas de
     `page_view` pur) — sous-compte le haut du funnel (numérateur de
     `taux_lancement` faussé).
   - **Reco : (a)** — le haut du funnel (visiteurs uniques) est précisément
     l'inconnue #1 du FORECAST ; le surcoût de réveil est négligeable.

6. **Rate-limiter aussi `/events` (et `/feedback`) ?**
   - Options : (a) **oui, léger** — un nouvel endpoint d'écriture public sans
     limite recrée la surface d'abus qu'on ferme sur `/analyze` (spam d'INSERT) ;
     (b) non — `/events` est cheap (pas de coût LLM), risque faible.
   - **Reco : (a) un rate-limit large** réutilisant la même brique B (ex. 60
     req/min/IP), pour ne pas laisser une porte ouverte. À cadrer dans le lot
     rate-limit.

7. **`feedback_submitted` : on le route via `/events` ou on garde `/feedback`
   seul ?**
   - Options : (a) **garder `/feedback`** et lire la table `feedback` pour ce
     signal (zéro duplication) ; (b) émettre aussi un event léger (deux sources de
     vérité) ; (c) tout migrer dans `events` (casse 9.7, écarté).
   - **Reco : (a).** La donnée est déjà collectée par 9.7.

8. **Confirme-t-on que le rate-limit **ne remplace pas** l'usage limit hard
   OpenAI (§9.4) et qu'on pose ce dernier ?**
   - **Reco : poser l'usage limit hard OpenAI (5 min, action humaine)** en plus du
     rate-limit. Le rate-limit protège la dispo + lisse ; l'usage limit est le
     filet financier dur. Les deux, pas l'un OU l'autre.

9. **`analysis_failed(reason=llm_fallback)` : on accepte qu'il soit mesuré côté
   serveur (le seul qui connaît le fallback) et pas côté client ?**
   - Options : (a) **oui** — event serveur sur le point unique du fallback
     (cohérent avec 9.3), le client ne gère que `url_unreachable` / `no_input` ;
     (b) faire remonter un flag `degraded` dans `AnalyzeResponse` → **change le
     contrat `/analyze`** (anti-pattern §11.9), refusé pour ce lot.
   - **Reco : (a).**

10. **Séquencement : scinder (rate-limit d'abord = 9.9, events ensuite = 9.10) ou
    lot combiné ?**
    - **Reco : scinder.** Le garde-fou (rate-limit, court, sans question RGPD
      ouverte) ne doit pas attendre que les arbitrages events (Q1-Q3, Q5, Q7)
      soient tranchés. Livrer le rate-limit d'abord sécurise `/analyze` **avant**
      d'ouvrir le nouvel endpoint public `/events`.
