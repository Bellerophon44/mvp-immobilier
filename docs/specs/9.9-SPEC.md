## 1. Objectif et perimetre

### Objectif
Proteger `POST /analyze` (cout LLM + disponibilite) et l'endpoint d'ecriture
public `POST /feedback` contre un abus en rafale, via une brique de rate-limit en
memoire (fenetre glissante, par IP), maison, sans nouvelle dependance, exposee
comme dependance FastAPI reutilisable.

### In
- Backend : une brique reutilisable de rate-limit en memoire (fenetre glissante,
  ~20 lignes), implementation maison, exposee comme dependance FastAPI
  parametrable (limite + duree de fenetre).
- Branchement sur `POST /analyze` : 10 requetes / 60 s / IP.
- Branchement sur `POST /feedback` : 60 requetes / 60 s / IP.
- Lecture de l'IP cliente via l'en-tete `Fly-Client-IP` (replis documentes en §3).
- Reponse `429` avec en-tete `Retry-After` au-dela du seuil.
- Tests : `backend/tests/test_rate_limit.py` + fixture de reinitialisation de
  l'etat memoire dans `backend/tests/conftest.py`.

### Out (non-objectifs, voir §5)
- Table `events`, endpoint `POST /events`, modele `EventIn`, emission front,
  cookie/session, beacon `methode_view` : tout cela est le lot **9.10**.
- `slowapi` ou toute nouvelle dependance Python (requirements.txt reste a 7 lignes).
- Usage limit hard OpenAI : action humaine deja actee, hors code (rappel en §6).
- Persistance du compteur (Redis / SQLite) : l'etat memoire mono-instance suffit
  (cf. §2.5).
- Rate-limit distribue multi-instance : non pertinent (1 VM Fly, cf. §2.5).
- Toute modification du schema `AnalyzeResponse`, de `AnalyzeRequest`, du contrat
  `/analyze` ou `/feedback` cote `frontend/lib/api.ts`.

## 2. Decisions actees (GATE 1)

1. **Implementation maison, zero dependance.** Brique de ~20 lignes : un
   `dict[str, deque[float]]` IP -> timestamps, purge des timestamps hors fenetre,
   comptage, `429` si depassement. **PAS de slowapi.** `requirements.txt` ne change
   pas.
2. **Brique reutilisable** exposee comme **dependance FastAPI** parametrable
   (limite, duree de fenetre), pour qu'un meme code serve `/analyze`, `/feedback`,
   et plus tard `/events` (9.10) sans duplication.
3. **IP via `Fly-Client-IP`.** Lecture en priorite de l'en-tete `Fly-Client-IP`
   (l'IP reelle derriere le proxy Fly). Replis dans l'ordre : `X-Forwarded-For`
   (premier hop) puis `request.client.host`. L'IP sert **uniquement** de cle de
   bucket en memoire transitoire : **jamais persistee, jamais journalisee.**
4. **`POST /analyze` : 10 req / 60 s / IP.** Au-dela : `429` + `Retry-After`.
5. **`POST /feedback` : 60 req / 60 s / IP** (limite plus large que `/analyze`,
   meme brique). `/events` n'existe pas encore : il sera rate-limite en 9.10 en
   reutilisant cette brique.
6. **Etat memoire mono-instance accepte.** 1 VM Fly auto-stop -> compteur process
   coherent. Perte de l'etat au restart/cold-start documentee comme **non
   bloquante** : une rafale maintient la VM eveillee, donc le compteur ne se vide
   pas pendant l'attaque (cf. analyse §1.5).

### Points non encore tranches (a remonter si besoin avant dev)
- **Corps de la reponse 429** : la valeur exacte du JSON d'erreur (ex.
  `{"detail": "..."}`) n'a pas ete imposee en GATE 1. Reco : `HTTPException(429,
  detail="Trop de requetes, reessayez plus tard.")`, message FR sans donnee
  technique ni IP. A confirmer si le front doit afficher un message specifique
  (sinon comportement par defaut accepte).
- **Cle de fenetre** : 60 s fixe. Si une granularite differente est voulue plus
  tard, la dependance etant parametrable, aucun changement de contrat n'est requis.

## 3. Contrat technique

### 3.1 Brique de rate-limit (nouveau module backend)
Module suggere : `backend/app/rate_limit.py` (emplacement indicatif). Doit exposer :

- Une **fabrique de dependance** parametrable, signature indicative :
  `rate_limiter(limit: int, window_seconds: float) -> Callable` renvoyant une
  dependance FastAPI utilisable via `Depends(...)`. La dependance recoit `Request`
  (pour lire les en-tetes) et, si depassement, leve
  `HTTPException(status_code=429, headers={"Retry-After": "<n>"})`.
- L'etat interne (`dict[str, deque[float]]`) doit etre **encapsule** et exposer un
  **point de reinitialisation** testable (ex. fonction `reset_rate_limit_state()`
  ou methode `.clear()`), pour la fixture d'isolation (cf. §3.5 et leçon 9.7).
- Une fonction interne de **resolution de l'IP** (`Fly-Client-IP` -> 1er hop
  `X-Forwarded-For` -> `request.client.host`). Cette fonction ne doit produire
  **aucun log** contenant l'IP.

Logger : si un logger nomme est introduit, il s'ajoute a la liste de
`backend/CLAUDE.md` §12 (loggers nommes). Il ne doit **jamais** journaliser l'IP
ni la valeur des en-tetes `Fly-Client-IP` / `X-Forwarded-For`. Un log d'evenement
`429` sans IP est acceptable (ex. compteur agrege, pas de cle).

### 3.2 Calcul de `Retry-After`
Valeur en **secondes entieres**, cohérente avec la fenetre glissante : duree
restante avant que le plus ancien timestamp de la fenetre n'expire (donc
`<= window_seconds`, et `>= 1`). Une valeur fixe egale a `window_seconds` (60) est
acceptable comme borne haute si le calcul exact n'est pas retenu, tant que la
valeur est **presente, numerique et positive**.

### 3.3 Endpoints impactes (signatures)
- `POST /analyze` : ajout de `Depends(rate_limiter(limit=10, window_seconds=60))`
  (ou equivalent) au handler `analyze` (`main.py:260`). **Aucun** changement de
  `AnalyzeRequest`, `AnalyzeResponse`, ni du corps/headers de la reponse 200.
- `POST /feedback` : ajout de `Depends(rate_limiter(limit=60, window_seconds=60))`
  au handler `submit_feedback` (`main.py:308`). **Aucun** changement de
  `FeedbackIn` ni de la reponse 201 `{"status": "ok"}`.

La dependance s'execute **avant** la logique metier (court-circuit `429` sans
appel LLM ni ecriture DB).

### 3.4 Schema DB / migrations / secrets
- **Aucune** colonne, table ou migration. Le rate-limit est purement en memoire.
- **Aucun** nouveau secret, variable d'env, ou ligne dans `requirements.txt`.
- **Aucun** changement de `frontend/lib/api.ts` (le front ne connait pas le 429 ;
  cote 9.9 on ne modifie pas le contrat front — un eventuel affichage UX du 429
  serait un sujet 9.10/UX distinct).

### 3.5 Isolation des tests (leçon 9.7)
Le compteur en memoire est un **etat partage de module** : sans reinitialisation,
les tests s'influencent (un test qui sature un bucket fait echouer le suivant).
Garde-fou requis :
- `backend/tests/conftest.py` : fixture (autouse, scope function) qui appelle le
  point de reinitialisation de §3.1 **avant chaque test**, pour repartir d'un etat
  vide. Ne jamais dependre de l'ordre des tests ni d'un `count()` global.
- Les tests injectent l'IP via l'en-tete HTTP `Fly-Client-IP` (TestClient :
  `client.post(..., headers={"Fly-Client-IP": "203.0.113.7"})`), jamais via l'IP
  reelle du `TestClient`.
- Pour tester la fenetre glissante **sans attendre 60 s reelles** : utiliser une
  **fenetre courte injectable** (la dependance etant parametrable) ou une horloge
  mockable. Reco : exposer la fenetre comme parametre et, pour le test de glissement,
  instancier une dependance dediee a fenetre tres courte (ex. 0,2 s) ou monkeypatcher
  la source de temps (`time.monotonic`). **Ne pas** faire de `time.sleep(60)`.

## 4. Criteres d'acceptation (testables, pytest)

Fichier cible : `backend/tests/test_rate_limit.py`. Valeurs exactes a tester.

1. **Sous le seuil `/analyze` ça passe.** Avec un body `/analyze` valide (ex.
   `{"raw_text": "..."}`) et une IP `Fly-Client-IP` fixe, les **10 premieres**
   requetes ne renvoient **pas** `429` (le code metier peut renvoyer 200/422/500
   selon le LLM mocke, mais **jamais** 429). Assertion ciblee : `resp.status_code
   != 429` pour les requetes 1 a 10.
2. **Au-dela du seuil `/analyze` -> 429.** La **11e** requete avec la meme IP
   dans la meme fenetre renvoie `status_code == 429`.
3. **En-tete `Retry-After` present et coherent.** La reponse `429` porte un
   en-tete `Retry-After` ; sa valeur est un entier `>= 1` et `<= 60`.
   Assertion : `"retry-after" in {k.lower() for k in resp.headers}` et
   `1 <= int(resp.headers["retry-after"]) <= 60`.
4. **Buckets par IP distincts.** Apres avoir sature `/analyze` depuis l'IP A
   (11e -> 429), une requete depuis l'IP B (`Fly-Client-IP` different) ne renvoie
   **pas** `429`. Deux IP differentes ne partagent pas le compteur.
5. **La fenetre glisse.** Avec une dependance a fenetre courte (ou horloge
   mockee), apres saturation puis expiration de la fenetre, une nouvelle requete
   de la meme IP repasse (status `!= 429`). Atteste que les timestamps anciens
   sont purges et que le rate-limit n'est pas un blocage definitif.
6. **`/feedback` a une limite plus large que `/analyze`.** Sur `/feedback`
   (`{"rating": 3}`), les **11e a 60e** requetes d'une meme IP ne renvoient **pas**
   `429` (la limite stricte de 10 d'`/analyze` ne s'y applique pas) ; la **61e**
   renvoie `429`. Assertion minimale falsifiable : la 11e requete `/feedback` est
   `!= 429` ET il existe un rang (`<= 61`) ou `/feedback` finit par renvoyer `429`.
7. **L'IP n'apparait dans aucun log.** Avec `caplog` au niveau DEBUG, apres une
   serie de requetes (dont une `429`) portant `Fly-Client-IP: 203.0.113.7`,
   l'assertion `"203.0.113.7" not in caplog.text` tient. Idem pour une valeur
   `X-Forwarded-For` injectee.
8. **Aucune nouvelle dependance.** Test lisant `backend/requirements.txt` :
   l'ensemble des packages est exactement
   `{fastapi, uvicorn[standard], sqlalchemy, requests, beautifulsoup4, openai,
   numpy}` (comparaison sur les noms normalises) et **ne contient pas** `slowapi`.
9. **Contrat `/analyze` inchange (statut metier preserve).** Une requete
   `/analyze` avec body `{}` (sous le seuil) renvoie toujours `400` (cf.
   `test_analyze_contract_unchanged`), pas `429` ni `500`. Le rate-limit ne
   modifie pas les codes metier existants.
10. **Contrat `/feedback` inchange.** Une requete `/feedback` `{"rating": 3}` sous
    le seuil renvoie toujours `201` et `{"status": "ok"}`. `{"rating": 6}` renvoie
    toujours `422`. Le rate-limit ne s'interpose pas sur la validation Pydantic.
11. **Repli d'IP fonctionnel.** Sans `Fly-Client-IP` mais avec
    `X-Forwarded-For: 198.51.100.5, 10.0.0.1`, le bucket est celui du **premier
    hop** (`198.51.100.5`) : saturer cette Iify la limite et une autre valeur de
    premier hop n'est pas bloquee. (Critere falsifiable du repli ; sans aucun
    en-tete, `request.client.host` du TestClient sert de cle, ce qui ne doit pas
    crasher.)
12. **Reinitialisation entre tests effective.** Un test qui sature un bucket
    n'impacte pas un test independant suivant (la fixture autouse de §3.5 garantit
    l'etat vide). Falsifiable : deux tests consecutifs saturant la **meme** IP
    passent tous les deux leur 1re requete `!= 429`.

## 5. Hors-perimetre / non-objectifs (rappel borne dev)
- Table `events`, `POST /events`, emission front, beacons, cookie/session : lot
  9.10.
- Persistance/partage du compteur (Redis, SQLite, multi-instance).
- Affichage UX du `429` cote front / modification de `frontend/lib/api.ts`.
- Usage limit hard OpenAI (action humaine, hors code).
- Rate-limit sur les endpoints `/admin/*` (proteges par token, hors scope abus
  public).

## 6. Conformite (anti-patterns applicables)
- **RGPD / minimisation** : l'IP est une donnee personnelle. Elle ne vit qu'en
  cle de dict memoire transitoire, **jamais ecrite en DB, jamais journalisee**,
  purgee par la fenetre glissante. Ligne rouge verrouillee par le critere 7.
- **Contrat `/analyze` stable** (invariant `lessons.md` ligne 19) : ni
  `AnalyzeRequest`, ni `AnalyzeResponse`, ni les codes metier ne changent ; le
  rate-limit est une dependance en amont. Verrouille par criteres 9 et 10.
- **Pas de nouvelle dependance / pas de vendor** : implementation maison,
  `requirements.txt` intact. Verrouille par critere 8.
- **Pas de secret en clair** : aucun secret introduit.
- **Pas d'estimation de prix / DVF / redistribution d'annonce** : sans objet (pas
  de logique produit touchee).
- **Conventions §12** : Python 3.12, logger nomme (sans IP), pas de commentaire
  "what", pas d'emoji.
- **Isolation des tests (leçon 9.7)** : etat memoire reinitialisable par fixture,
  pas d'etat partage entre tests, jamais de `count()` global ni de dependance a
  l'ordre. Verrouille par critere 12.
